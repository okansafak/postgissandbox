## 💡 Temel Kavramlar

### 1. PostgreSQL High Availability (Streaming Replication + Failover)

**Tanım:** Primary database'in write işlemlerini replica(lar)a real-time olarak aktaran ve primary başarısız olduğunda replica'nın üstü almıştır çalışması.

**Uygulama:**
- **Primary (İstanbul-DC1):** 16-core PostgreSQL server, WAL (Write-Ahead Log) stream'i 2 replica'ya gönderir
- **Replica-1 (İstanbul-DC2):** Standby mode, replication client ile WAL alır, kalıp XLOG apply eder
- **Replica-2 (Backup-DC):** Ek replica, failover senaryolarında reserve
- **Pgpool-II (pooler + failover):** Primary'den replica'ya write/read split, primary başarısızlığı → otomatik replica promote
- **Recovery timeline:** Primary çöküş → pgpool detection (5-10 sn) → replica promote (10-15 sn) → application reconnect (5 sn) = ~30 sn RTO

**Performans ipucu:** Streaming replication'da apply lag 0-100ms tutmak için replica'nın read workload'u minimize et.

### 2. Connection Pooling (pgBouncer)

**Tanım:** Veritabanı bağlantılarının havuzlanması. Şu problem çözer: Bağlantı açılması/kapatılması OS system call → overhead.

**Uygulama:**
- **pgBouncer proxy:** 1000 client connection → pgBouncer → 50 PostgreSQL server connection (multiplexing)
- Mode: `transaction` (her transaction sonunda bağlantı pool'a döner) vs `session` (client session boyunca bind kalır)
- **Rate limiting:** Max 100 connection/second → excess queue, retry
- **Performance:** Bağlantı pool olmadan: 100ms setup/teardown × 10K transaction/sn = 1000 sn CPU waste/sn. Pool ile: 1ms × 10K = 10 sn waste (99% iyileşme)

### 3. Logical Replication (Change Data Capture)

**Tanım:** PostgreSQL WAL'dan data changes'ı (INSERT/UPDATE/DELETE) logical format'da çıkarmak ve downstream system'lere (analytics DB, cache invalidation, mobile sync) iletmek.

**Uygulama:**
- **Publication (source):** `CREATE PUBLICATION analytics_pub FOR TABLE addresses, eczanes, ...`
- **Subscription (destination):** Analytics database'de `CREATE SUBSCRIPTION ... CONNECT TO ... PUBLICATION analytics_pub`
- **CDC delay:** <100ms (primary'den analytics DB'ye)
- **Use case:** İstanbul'un tüm adreslerini real-time analytics cluster'a replike et

**Performans ipucu:** Subscription'da slot_name uniqueness → multiple consumer'lar same topic consume edebilir (fan-out pattern).

### 4. Disaster Recovery: Backup Strategy (RPO vs RTO)

**Tanım:**
- **RPO (Recovery Point Objective):** Maksimum kabul edilir data loss. Örn., "RPO=1 hour"
- **RTO (Recovery Time Objective):** Maksimum kabul edilir downtime. Örn., "RTO=4 hours"
- **Target:** RPO=1 hour, RTO=4 hours → WAL archiving + daily backup

**Uygulama:**
- **Full backup:** Her hafta Pazarı `pg_dump` + gzip → S3
- **Incremental backup (WAL archiving):** Her WAL segment (16 MB) S3'e archive edilir
- **PITR (Point-In-Time Recovery):** "2026-04-23 14:32:15'teki tablo snapshot'ını geri al"
- **Restore test:** Aylık dry-run (backup'dan production replica'da restore et)

**Performans ipucu:** WAL archiving'de lz4 compression (hızlı, 60-70% ratio) vs gzip (yavaş, 80% ratio) trade-off.

### 5. Monitoring & Alerting (Prometheus + Grafana)

**Tanım:** Production system'in metrikleri (CPU, memory, query latency, replication lag) topla, threshold'lar set et, ihlallerde notification gönder.

**Uygulama:**
- **Metrics collection:** Prometheus PostgreSQL exporter her 15 saniyede PostgreSQL'den metrikleri çeker
- **Metric examples:**
  - `pg_stat_replication_wal_lsn_diff` = Primary vs replica WAL position farkı. > 100 MB → "Replica lag high" alert
  - `pg_stat_statements_mean_exec_time` = Ortalama query execution zamanı. > 1000 ms → "Slow queries detected"
  - `pg_database_size_bytes` = Veritabanı boyutu. > 500 GB → "Disk space warning"
- **Grafana dashboard:** 12-panel dashboard (CPU trend, memory usage, query latency histogram, replication lag)
- **Alert routing:** PagerDuty/Slack:
  - **Critical:** Replica lag > 500MB → SMS + immediate page
  - **Warning:** Query latency p95 > 500ms → Slack message
  - **Info:** Backup completed → Slack notification

### 6. Zero-Downtime Deployment (Blue-Green, Canary)

**Tanım:** Yeni kod version'ını dağıtırken hiçbir user'ı affect etmeden smooth transition.

**Uygulama (Blue-Green):**
- **Blue:** Mevcut production (v1.2.3)
- **Green:** Yeni version (v1.2.4), staging'de test edilmiş
- **Deployment:**
  1. Green (v1.2.4) pod'larını spin up et, warmup (cache, connection pool)
  2. Load balancer'da trafik % 0 Green, % 100 Blue
  3. Health check'lar pass → % 100 Green'e switch (1 saniye)
  4. Monitoring 5 dakika → error rate > 1% → % 100 Blue'ye rollback

**Kubernetes integration:** Deployment manifest ile pod'ları define et, livenessProbe + readinessProbe ile health check'lar set et.

1. **Streaming Replication + Pgpool:** Primary-replica replication < 10ms lag + automatic failover < 30sn ile RTO/RPO iyileştirilir.

6. **Disaster Recovery Drill:** Monthly restore test (backup → test DB) confidence build.

7. **Security in Replication:** Replication channel SSL/TLS encrypted. Replica read-only mode. WAL archiving S3-encrypted.

8. **Cost Optimization:** WAL archiving lz4 (fast compression). S3 storage class GLACIER_IR. Connection pool → server count azaltma → cost reduction.
