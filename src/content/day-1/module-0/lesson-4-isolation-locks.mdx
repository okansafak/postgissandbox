## Öğrenim Hedefleri

Eşzamanlı transaction'ların birbirini nasıl etkilediği, veritabanı tutarlılığının en kritik boyutlarından biridir. Bu ders, SQL standardındaki isolation seviyelerini, PostgreSQL'in kilit mekanizmalarını ve deadlock senaryolarını uygulamalı olarak ele almaktadır.

- READ COMMITTED, REPEATABLE READ ve SERIALIZABLE farkını açıklayabilmek
- `pg_locks` ile aktif kilitleri gözlemlemek
- Deadlock senaryosunu tanımlamak ve önlem almak

---

## Isolation Seviyeleri

SQL standardı dört isolation seviyesi tanımlar. Her seviye, eşzamanlı transaction'lar arasında farklı görünürlük anomalilerine izin verir:

| Seviye | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | Mümkün* | Mümkün | Mümkün |
| **READ COMMITTED** | İmkânsız | Mümkün | Mümkün |
| REPEATABLE READ | İmkânsız | İmkânsız | Mümkün* |
| SERIALIZABLE | İmkânsız | İmkânsız | İmkânsız |

*PostgreSQL, READ UNCOMMITTED'ı READ COMMITTED gibi davrandırır. REPEATABLE READ ise phantom read'i de engeller — SQL standardının ötesinde bir güvence sağlar.*

PostgreSQL'in varsayılan isolation seviyesi **READ COMMITTED**'tır.

## Uygulama Örneği

### Mevcut isolation seviyesi

<RunnableBlock
  sql={`SHOW transaction_isolation;`}
  label="Transaction isolation seviyesi"
  description="SHOW transaction_isolation, aktif oturumun veya transaction'ın isolation seviyesini döndürür. Varsayılan değer 'read committed'tır; her SQL ifadesi başında yeni bir anlık görüntü alınır."
/>

### REPEATABLE READ snapshot

<RunnableBlock
  sql={`DROP TABLE IF EXISTS hesap;
CREATE TABLE hesap (
  id      SERIAL PRIMARY KEY,
  bakiye  NUMERIC
);
INSERT INTO hesap (bakiye) VALUES (1000), (2000);

BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT id, bakiye FROM hesap ORDER BY id;
COMMIT;

SELECT id, bakiye FROM hesap ORDER BY id;`}
  label="REPEATABLE READ snapshot"
  description="BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ ile açılan transaction, başlangıçtaki veri görüntüsünü sabit tutar. Başka bir bağlantı aynı satırları güncelleyip commit etse bile bu transaction eski değerleri görmeye devam eder. Finansal raporlama senaryolarında tutarsız okumayı önler."
/>

### SERIALIZABLE ile güvenli transfer

<RunnableBlock
  sql={`BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  UPDATE hesap SET bakiye = bakiye - 100 WHERE id = 1;
  UPDATE hesap SET bakiye = bakiye + 100 WHERE id = 2;
COMMIT;

SELECT id, bakiye FROM hesap ORDER BY id;`}
  label="SERIALIZABLE transfer"
  description="SERIALIZABLE, tüm eşzamanlı transaction'ların sanki sırayla çalışmış gibi sonuç üretmesini zorunlu kılar. İki transaction aynı satırları çakışan biçimde okuyup yazarsa biri 'serialization failure' hatası alır ve yeniden denenmesi gerekir. En güçlü tutarlılık garantisini sağlar."
/>

### Aktif kilitleri gözlemle

<RunnableBlock
  sql={`SELECT
  locktype,
  relation::regclass AS tablo,
  mode,
  granted
FROM pg_locks
WHERE relation IS NOT NULL
ORDER BY granted DESC, mode;`}
  label="Aktif kilitler"
  description="pg_locks görünümü, cluster genelindeki tüm aktif kilitleri listeler. 'granted = true' kilit onaylandı demektir; 'false' ise başka bir transaction'ın kilit bırakmasını beklediğini gösterir. Tek bağlantılı bu ortamda kilit sayısı azdır; çok kullanıcılı ortamlarda tıkanıklıkları teşhis etmek için kritiktir."
/>

### SELECT FOR UPDATE — satır kilidi

<RunnableBlock
  sql={`BEGIN;
  SELECT id, bakiye
  FROM hesap
  WHERE id = 1
  FOR UPDATE;
  UPDATE hesap SET bakiye = bakiye - 50 WHERE id = 1;
COMMIT;

SELECT id, bakiye FROM hesap ORDER BY id;`}
  label="SELECT FOR UPDATE"
  description="FOR UPDATE ifadesi seçilen satırlara exclusive (dışlayıcı) kilit koyar. Bu transaction tamamlanana kadar başka bir transaction aynı satırı güncelleyemez veya kilitleyemez. Önce oku — hesapla — sonra yaz kalıplarında tutarsız güncellemeyi (lost update) önler."
/>

### Advisory lock — uygulama seviyesi kilit

<RunnableBlock
  sql={`SELECT pg_try_advisory_lock(12345) AS kilit_alindi;

-- İşlemler burada...

SELECT pg_advisory_unlock(12345) AS serbest_birakildi;`}
  label="Advisory lock"
  description="pg_try_advisory_lock, uygulama tanımlı sayısal bir kilit almayı dener; kilit alınabildiyse true, başkası tutuyorsa false döner. Bu kilitler tablo satırlarına değil, sayısal bir token'a bağlıdır; bağlantı kapandığında otomatik serbest kalır. Uygulama düzeyinde iş akışı kilitleme için kullanılır."
/>

## Deadlock

Deadlock, iki transaction'ın birbirinin kilitlediği kaynağı beklediği kilitlenme durumudur:

```
T1: LOCK hesap(id=1)  →  bekliyor hesap(id=2)
T2: LOCK hesap(id=2)  →  bekliyor hesap(id=1)
```

PostgreSQL, deadlock'u `lock_timeout` süresi dolmadan otomatik algılar ve transaction'lardan birini `ERROR: deadlock detected` hatasıyla sonlandırır.

**Temel önlem:** Birden fazla satıra kilit koyulacaksa her zaman aynı deterministik sırayla (örneğin artan `id`) kilitleme yapılmalıdır.

---

## Özet

- Varsayılan isolation seviyesi **READ COMMITTED** — her ifade başında yeni snapshot alınır.
- **REPEATABLE READ**, finansal raporlar ve uzun süreli okuma işlemleri için idealdir.
- **SERIALIZABLE** en güçlü tutarlılığı sağlar; ancak çakışma oranı yüksek iş yüklerinde yeniden deneme mantığı gerektirir.
- Deadlock'tan kaçınmak için kilitleme sırası her zaman tutarlı olmalıdır.
