## Öğrenim Hedefleri

PostgreSQL'in eşzamanlılık mimarisinin temeli olan MVCC (Multi-Version Concurrency Control) mekanizması, okuyucuların yazarları — yazarların okuyucuları beklemesini ortadan kaldırır. Bu ders MVCC'nin nasıl çalıştığını ve transaction komutlarının nasıl kullanıldığını uygulamalı olarak ele almaktadır.

- MVCC'nin ölü satır sorununu nasıl çözdüğünü açıklayabilmek
- `BEGIN / COMMIT / ROLLBACK` ile transaction yazmak
- `xmin` ve `xmax` sistem sütunlarını sorgulamak

---

## MVCC Nedir?

Geleneksel veritabanları okuma-yazma çakışmasını satır kilitleriyle çözer; okuyucular yazarları, yazarlar okuyucuları bekler. PostgreSQL farklı bir model uygular: her `UPDATE` veya `DELETE` işlemi eski satırı silmez, yeni bir sürüm ekler. Eski sürüm "ölü satır" (dead tuple) olarak tabloda kalır ve VACUUM tarafından geri kazanılır.

```
Satır v1 (xmin=100, xmax=200)  ← Transaction 200 bu satırı güncelledi
Satır v2 (xmin=200, xmax=NULL) ← Aktif (görünür) sürüm
```

Bu yaklaşım şu garantileri sağlar:

- Okuyucular hiçbir zaman yazarı beklemez.
- Her transaction, başladığı andaki tutarlı "anlık görüntü"yü (snapshot) görür.

## Uygulama Örneği

### xmin / xmax sistem sütunları

<RunnableBlock
  sql={`DROP TABLE IF EXISTS mvcc_test;
CREATE TABLE mvcc_test (
  id   SERIAL PRIMARY KEY,
  veri TEXT
);

INSERT INTO mvcc_test (veri) VALUES ('ilk satır'), ('ikinci satır');

SELECT id, veri, xmin, xmax
FROM mvcc_test;`}
  label="xmin/xmax görüntüle"
  description="xmin, satırı tabloya ekleyen transaction ID'sini; xmax, satırı silen veya güncelleyen transaction ID'sini saklar. xmax = 0 (ya da NULL olarak görünmesi) satırın hâlâ aktif olduğunu gösterir. Bu iki sütun MVCC'nin çalışma izini doğrudan yansıtır."
/>

### BEGIN / COMMIT ile transaction

<RunnableBlock
  sql={`BEGIN;
  INSERT INTO mvcc_test (veri) VALUES ('transaction satır 1');
  INSERT INTO mvcc_test (veri) VALUES ('transaction satır 2');
COMMIT;

SELECT id, veri FROM mvcc_test ORDER BY id;`}
  label="Başarılı transaction"
  description="BEGIN ile başlatılan transaction içindeki tüm INSERT işlemleri atomik bir birim olarak gerçekleşir. COMMIT ile her iki satır da kalıcı olarak yazılır; hata durumunda ikisi birden geri alınırdı."
/>

### ROLLBACK ile geri alma

<RunnableBlock
  sql={`BEGIN;
  INSERT INTO mvcc_test (veri) VALUES ('silinecek satır');
ROLLBACK;

SELECT count(*) AS satir_sayisi FROM mvcc_test;`}
  label="ROLLBACK ile geri alma"
  description="ROLLBACK, transaction içinde yapılan tüm değişiklikleri geri alır. INSERT edilen satır tabloya hiç yazılmamış gibi davranılır; COUNT sorgusu bu satırı saymaz."
/>

### SAVEPOINT ile kısmi geri alma

<RunnableBlock
  sql={`BEGIN;
  INSERT INTO mvcc_test (veri) VALUES ('iyi satır');
  SAVEPOINT sp1;
  INSERT INTO mvcc_test (veri) VALUES ('kötü satır');
  ROLLBACK TO SAVEPOINT sp1;
COMMIT;

SELECT veri FROM mvcc_test ORDER BY id DESC LIMIT 3;`}
  label="SAVEPOINT kullanımı"
  description="SAVEPOINT, transaction içinde bir kontrol noktası oluşturur. ROLLBACK TO SAVEPOINT yalnızca bu noktadan sonraki değişiklikleri geri alır; öncekiler korunur. Burada 'kötü satır' geri alınır, 'iyi satır' COMMIT ile kalıcı hale gelir."
/>

### Dead tuple gözlemi

<RunnableBlock
  sql={`UPDATE mvcc_test SET veri = 'güncellendi' WHERE id = 1;

SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
WHERE relname = 'mvcc_test';`}
  label="Dead tuple gözlemi"
  description="Her UPDATE işlemi eski satırı 'ölü tuple' olarak bırakır. pg_stat_user_tables'daki n_dead_tup sütunu, VACUUM'un henüz geri kazanmadığı eski sürümlerin sayısını gösterir. Bu değer yüksekse VACUUM beklemede demektir."
/>

## Özet

| Komut | İşlev |
|---|---|
| `BEGIN` | Transaction başlatır |
| `COMMIT` | Değişiklikleri kalıcı yazar |
| `ROLLBACK` | Tüm değişiklikleri geri alır |
| `SAVEPOINT ad` | Transaction içinde kontrol noktası oluşturur |
| `ROLLBACK TO SAVEPOINT` | Belirtilen noktaya kadar geri alır |
