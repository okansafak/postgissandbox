## 🎯 Ne öğreneceksin?

- Proje gereksinimlerine göre koordinat sistemi seçim kararı verebileceksin
- Depolama, hesaplama ve görüntüleme için hangi SRID'in uygun olduğunu belirleyebileceksin
- Bu modülde öğrenilen tüm kavramları bir arada kullanabileceksin

---

## 📖 Kavram: Karar Ağacı

```
Veri nasıl kullanılacak?
│
├── Dünya/ülke geneli veri → 4326 (WGS84) + geography cast
│
├── Web haritasına tile olarak aktarılacak → 3857 (Web Mercator)
│
├── Hassas alan/mesafe ölçümü (Türkiye)
│   ├── İstanbul/İzmir bölgesi → UTM 35N (32635)
│   ├── Ankara/Konya bölgesi  → UTM 36N (32636)
│   └── Doğu Türkiye          → UTM 37N (32637)
│
└── Kadastro / resmi mühendislik → ITRF96 (5254-5259)
```

---

## 📖 Depolama vs Hesaplama vs Görüntüleme

| Aşama | Önerilen SRID | Neden |
|---|---|---|
| **Depolama** | 4326 | Evrensel, GeoJSON uyumlu, indeks dostlu |
| **Mesafe/alan hesabı** | `::geography` veya UTM | Metre cinsinden doğru sonuç |
| **Web görüntüleme** | 3857 (tile) ya da 4326 (GeoJSON) | Harita kütüphanesi standardı |
| **Raporlama / kadastro** | UTM veya ITRF96 | Yerel hassasiyet |

**Altın kural:** Veriyi 4326'da depola, hesabı geography veya uygun UTM'de yap.

## ▶ Canlı Örnek — Tam Akış: Depola → Hesapla → Göster

<RunnableBlock
  sql={`-- 1. Tablo oluştur: 4326'da depola
CREATE TABLE IF NOT EXISTS noktalar_demo (
  id   SERIAL PRIMARY KEY,
  ad   TEXT,
  geom GEOMETRY(POINT, 4326)
);

-- 2. Veri yükle
INSERT INTO noktalar_demo (ad, geom) VALUES
  ('İstanbul', ST_SetSRID(ST_MakePoint(28.979, 41.015), 4326)),
  ('Ankara',   ST_SetSRID(ST_MakePoint(32.854, 39.930), 4326)),
  ('İzmir',    ST_SetSRID(ST_MakePoint(27.143, 38.423), 4326))
ON CONFLICT DO NOTHING;

-- 3. Sorgula: mesafeyi geography ile hesapla, haritada 4326 göster
SELECT
  a.ad AS kaynak,
  b.ad AS hedef,
  ROUND(ST_Distance(a.geom::geography, b.geom::geography)::numeric / 1000, 0) AS mesafe_km,
  a.geom  -- harita için 4326
FROM noktalar_demo a
CROSS JOIN noktalar_demo b
WHERE a.id < b.id
ORDER BY mesafe_km;`}
  label="Tam akış: depola + hesapla + göster"
/>

## ▶ Egzersiz: UTM vs geography Karşılaştırması

<RunnableBlock
  sql={`-- İzmir - İstanbul mesafesi: geography vs UTM 35N
SELECT
  'geography (elipsoid)' AS yontem,
  ROUND(ST_Distance(
    ST_SetSRID(ST_MakePoint(27.143, 38.423), 4326)::geography,
    ST_SetSRID(ST_MakePoint(28.979, 41.015), 4326)::geography
  )::numeric / 1000, 1) AS km
UNION ALL
SELECT
  'UTM 35N (planar)',
  ROUND(ST_Distance(
    ST_Transform(ST_SetSRID(ST_MakePoint(27.143, 38.423), 4326), 32635),
    ST_Transform(ST_SetSRID(ST_MakePoint(28.979, 41.015), 4326), 32635)
  )::numeric / 1000, 1);`}
  label="geography vs UTM mesafe karşılaştırması"
/>

> İki yöntem ~0.1 km fark vermelidir — her ikisi de kabul edilebilir doğrulukta.

## 🧪 Dene — Bölge Tespiti

Koordinata göre hangi UTM dilimini kullanman gerektiğini söyleyen bir sorgu yaz:

<RunnableBlock
  sql={`WITH test_noktalari AS (
  SELECT 'Trabzon'  AS sehir, 39.726 AS lon UNION ALL
  SELECT 'Diyarbakır', 40.228            UNION ALL
  SELECT 'Konya',    32.483             UNION ALL
  SELECT 'Çanakkale', 26.413
)
SELECT
  sehir,
  lon,
  CASE
    WHEN lon BETWEEN 24 AND 30 THEN 'UTM 35N (32635)'
    WHEN lon BETWEEN 30 AND 36 THEN 'UTM 36N (32636)'
    WHEN lon BETWEEN 36 AND 42 THEN 'UTM 37N (32637)'
    ELSE 'Sınır bölge — kontrol et'
  END AS onerilен_utm
FROM test_noktalari
ORDER BY lon;`}
  label="Koordinata göre UTM dilim önerisi"
/>

## ⚠ Modül Özeti — Kritik Kurallar

1. **Ham koordinatı depolarken:** `ST_SetSRID(ST_MakePoint(lon, lat), 4326)` — her zaman 4326 etiketle.
2. **Dönüştürmek için:** `ST_Transform(geom, hedef_srid)` — kaynak SRID bilinmeli.
3. **Mesafe/alan:** `::geography` veya UTM `ST_Transform` — asla SRID'siz `ST_Distance`.
4. **Depolama formatı:** `geometry(POINT, 4326)` kolonu — UTM değil, 4326.
5. **SetSRID ≠ Transform:** SetSRID etiketler, Transform dönüştürür.
