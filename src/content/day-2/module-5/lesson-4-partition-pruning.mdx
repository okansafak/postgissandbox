## 📖 Kavram: Partition Pruning

Planner sorgu WHERE koşulunu partition sınırlarıyla karşılaştırır:
- Eşleşmeyen partition'lar plan dışında bırakılır (**static pruning**)
- Çalışma zamanında bilinen parametreler için de çalışır (**runtime pruning**)

```
EXPLAIN
SELECT * FROM gps_kayit WHERE zaman >= '2024-01-01' AND zaman < '2024-02-01';

→ Bitmap Heap Scan on gps_kayit_2024_01  ← sadece bu
   (gps_kayit_2024_02 ve _03 planda YOK)
```

---

## 📖 Pruning Çalışmadığında

Pruning, partition key **doğrudan** karşılaştırılıyorsa çalışır:

| Sorgu | Pruning? |
|---|---|
| `WHERE zaman = '2024-01-15'` | ✅ Evet |
| `WHERE zaman::date = '2024-01-15'` | ❌ Hayır (fonksiyon var) |
| `WHERE sehir = 'istanbul'` | ✅ Evet |
| `WHERE UPPER(sehir) = 'ISTANBUL'` | ❌ Hayır |

## ▶ Her İki Filtre Birlikte — Optimum Plan

<RunnableBlock
  sql={`-- Zaman partition pruning + mekan index
-- (gps_kayit_2024_01 üzerinde idx_gps_kayit_01_geom olmalı)
EXPLAIN ANALYZE
SELECT arac, COUNT(*) AS kayit
FROM gps_kayit
WHERE zaman >= '2024-01-01' AND zaman < '2024-02-01'
  AND ST_DWithin(
    geom::geography,
    ST_SetSRID(ST_MakePoint(29.1, 41.0),4326)::geography,
    30000)
GROUP BY arac
ORDER BY kayit DESC;`}
  label="Pruning + GIST indeks birlikte"
/>

## ⚠ Tuzaklar

1. **Partition sayısı arttıkça plan oluşturma süresi artar:** 1000 partition'lı tablo için planner overhead önemli olabilir. PG 12+ ile iyileştirildi.
2. **`pg_partman` extension:** Otomatik partition oluşturma ve eski partition'ları arşivleme için. PGlite'ta mevcut değil.
3. **Global index yok:** PostgreSQL global (cross-partition) unique constraint/index oluşturulamaz (PK hariç, o da partition key içeriyorsa). Tasarımda göz önünde bulundur.
