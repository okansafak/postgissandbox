## Öğrenim Hedefleri

Metin biçimindeki EXPLAIN çıktısı insan tarafından okunabilir; ancak programatik analiz için JSON formatı daha uygundur. Bu ders, JSON plan formatını nasıl okuyup temel metrikleri nasıl çıkaracağını ve hangi node'un darboğaz oluşturduğunu nasıl bulacağını ele almaktadır.

- `EXPLAIN (FORMAT JSON)` çıktısını parse ederek temel metrikleri çıkarmak
- Hangi node'un en çok zaman harcadığını bulmak
- `Planning Time` ve `Execution Time` değerlerini yorumlamak

---

## JSON Plan Formatı

`FORMAT JSON` seçeneği plan hiyerarşisini yapısal bir JSON nesnesi olarak döndürür:

```json
[{
  "Plan": {
    "Node Type": "Aggregate",
    "Startup Cost": 0.00,
    "Total Cost": 1234.56,
    "Plan Rows": 1,
    "Actual Total Time": 45.2,
    "Plans": [{
      "Node Type": "Index Scan",
      ...
    }]
  },
  "Planning Time": 0.5,
  "Execution Time": 45.8
}]
```

Bu format `explain.depesz.com`, pgAdmin ve DBeaver gibi araçlara yapıştırılarak görselleştirilebilir.

### JSON planından metrik çıkarma

<RunnableBlock
  sql={`-- JSON planı parse et ve temel metrikleri göster
WITH plan_data AS (
  SELECT (EXPLAIN (ANALYZE, FORMAT JSON)
    SELECT COUNT(*) FROM noktalar_perf
    WHERE ST_DWithin(
      geom::geography,
      ST_SetSRID(ST_MakePoint(28.979,41.015),4326)::geography,
      150000)
  )[0] AS plan
)
SELECT
  plan->'Plan'->>'Node Type'             AS kok_node,
  (plan->'Plan'->>'Total Cost')::numeric AS toplam_maliyet,
  plan->>'Planning Time'                 AS planlama_ms,
  plan->>'Execution Time'                AS yurutme_ms
FROM plan_data;`}
  label="JSON planından metrik çıkarma"
  description="WITH CTE içinde EXPLAIN JSON sonucu saklanır; ardından JSON operatörleriyle temel metrikler çıkarılır. ->'Plan' JSON nesnesini, ->>'Node Type' metin değerini döndürür. [0] JSON dizisinin ilk (ve tek) elemanını seçer. Planlama süresi (Planning Time) bazen yürütme süresinden (Execution Time) uzun olabilir; bu durum çok karmaşık sorgu şemalarında görülür."
/>

## Pratik Uygulama

<RunnableBlock
  sql={`-- Planlama süresi bazen execution'dan uzun olabilir
EXPLAIN (ANALYZE, FORMAT JSON)
SELECT
  a.id,
  ST_AsText(a.geom) AS wkt,
  COUNT(b.id) AS yakin_nokta
FROM noktalar_perf a
JOIN noktalar_perf b
  ON ST_DWithin(a.geom::geography, b.geom::geography, 1000)
WHERE a.id BETWEEN 1 AND 5
GROUP BY a.id, a.geom;`}
  label="Planlama süresi ölçümü"
  description="Aynı tablonun kendisiyle ST_DWithin join'i yapılan karmaşık sorgunun JSON planı incelenir. 'Planning Time' ve 'Execution Time' değerleri karşılaştırılır. a.id BETWEEN 1 AND 5 yalnızca 5 satırı dış taraf olarak kullandığından arama alanı sınırlıdır; ancak nested loop join iç tarafı her dış satır için tekrar çalıştırır. Bu kalıp büyük tablolarda her zaman pahalıdır."
/>

## Yaygın Hatalar

1. **JSON plan `EXPLAIN` sonucunun ilk elemanıdır.** Sonuç bir JSON dizisidir — `[0]` ile ilk eleman alınmalıdır.
2. **`Actual Total Time` kümülatiftir.** Parent node, tüm child node sürelerini kapsayan toplam süreyi gösterir. Gerçek darboğazı bulmak için en uzun yaprak (leaf) node incelenmelidir.
3. **PGlite'ta JSON işlemleri sınırlı olabilir.** PGlite `->` ve `->>` JSON operatörlerini destekler; ancak `jsonb_path_query` gibi gelişmiş fonksiyonlar desteklenmeyebilir.
