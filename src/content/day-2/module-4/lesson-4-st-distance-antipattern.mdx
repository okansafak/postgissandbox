## Öğrenim Hedefleri

Mesafe filtresi yazmanın iki yolu vardır; biri GiST indeksini kullanır, diğeri kullanmaz. Bu fark büyük tablolarda sorgu süresini saniyeden milisaniyeye düşürebilir. Bu ders, mekansal sorgunun en yaygın performans antipattern'ini ve çözümünü ele almaktadır.

- `WHERE ST_Distance(...) < x` neden indeks kullanmadığını açıklamak
- `ST_DWithin` ile her zaman değiştirme gerekçesi
- Mesafe filtresinde `geography` ve `geometry` davranış farkı

---

## Antipattern: ST_Distance < N

```sql
-- YANLIŞ — indeksi bypass eder, her satırı hesaplar
WHERE ST_Distance(geom::geography, ref_point::geography) < 100000

-- DOĞRU — GiST indeksini kullanır
WHERE ST_DWithin(geom::geography, ref_point::geography, 100000)
```

**Neden?** `ST_Distance < x` bir fonksiyon sonucunu karşılaştırır — PostgreSQL bunu "sargable" (indeks uyumlu) görmez ve tam tablo taraması yapar. `ST_DWithin` ise GiST operatör sınıfı tarafından desteklenen özel bir operatör ailesidir ve indeks taramasına çevrilir.

---

## Geography vs Geometry Mesafe

| | `geometry` | `geography` |
|---|---|---|
| Mesafe birimi | Koordinat birimi (derece) | Metre |
| `ST_DWithin(a, b, 100000)` | 100.000 derece (tüm dünya!) | 100 km |
| Doğruluk | Düzlemsel | Elipsoid (gerçek dünya) |

**Kural:** Metre cinsinden mesafe filtresi için `::geography` cast kullanılmalıdır.

### ST_Distance antipattern planı

<RunnableBlock
  sql={`-- Hazırlık: indeksli tablo
CREATE TABLE IF NOT EXISTS mesafe_test (
  id   SERIAL PRIMARY KEY,
  geom GEOMETRY(POINT, 4326)
);

INSERT INTO mesafe_test (geom)
SELECT ST_SetSRID(ST_MakePoint(26+random()*19, 36+random()*6), 4326)
FROM generate_series(1, 3000)
ON CONFLICT DO NOTHING;

CREATE INDEX IF NOT EXISTS idx_mesafe_geom ON mesafe_test USING gist(geom);
ANALYZE mesafe_test;

-- YANLIŞ — planı gör
EXPLAIN ANALYZE
SELECT COUNT(*) FROM mesafe_test
WHERE ST_Distance(
  geom::geography,
  ST_SetSRID(ST_MakePoint(28.979,41.015),4326)::geography
) < 200000;`}

  description="3000 noktalı indeksli tabloda ST_Distance < 200000 koşuluyla sorgu EXPLAIN ANALYZE ile incelenir. 'actual time' ve 'rows' değerleri tüm tablonun tarandığını doğrular."

### ST_DWithin — doğru yol

<RunnableBlock
  sql={`-- DOĞRU — GiST indeksi kullanır
EXPLAIN ANALYZE
SELECT COUNT(*) FROM mesafe_test
WHERE ST_DWithin(
  geom::geography,
  ST_SetSRID(ST_MakePoint(28.979,41.015),4326)::geography,
  200000
);`}
  label="ST_DWithin — indeks kullanan plan"
  description="Aynı sorgu ST_DWithin ile yazıldığında EXPLAIN planı 'Index Scan' veya 'Bitmap Index Scan' gösterir. 'actual time' değeri antipattern sürümünden belirgin biçimde düşük olmalıdır. Bu karşılaştırma iki sorgu arasındaki pratik performans farkını somut sayılarla ortaya koyar."
/>

### KNN — en yakın N nokta

<RunnableBlock
  sql={`-- KNN: <-> operatörü GiST indeksi ile çalışır
SELECT
  id,
  ROUND(ST_Distance(
    geom::geography,
    ST_SetSRID(ST_MakePoint(28.979,41.015),4326)::geography
  )::numeric / 1000, 1) AS km
FROM mesafe_test
ORDER BY geom <-> ST_SetSRID(ST_MakePoint(28.979,41.015),4326)
LIMIT 5;`}
  label="KNN — en yakın 5 nokta"
  description="En yakın N nokta sorgusunda ORDER BY ST_Distance(...) tam tablo taraması yapar. Bunun yerine ORDER BY geom <-> ref GiST indeksini devreye sokar ve sadece en yakın adayları sıralar. ST_Distance SELECT'te doğrulama için kullanılır ancak sıralama için kullanılmaz. <-> operatörü bbox merkezine göre yaklaşık mesafe üretir; kesin mesafe ST_Distance ile doğrulanır."
/>

## Yaygın Hatalar

1. **`ST_Distance + ORDER BY + LIMIT` kombinasyonunda indeks seçilebilir.** `LIMIT` varsa planlayıcı bazen index scan'a geçebilir; ancak bu garanti değildir. KNN sorguları için `<->` operatörü her zaman tercih edilmelidir.
2. **`ST_DWithin(geometry, geometry, N)` derece kullanır.** `::geography` cast'ı olmadan N değeri metre değil derece olarak yorumlanır; istemeden büyük alanlar sorgulanabilir.
3. **Çok büyük `ST_DWithin` alanı.** 2000 km yarıçaplı bir sorgu neredeyse tüm tabloyu döndürür; bu durumda Seq Scan normal ve doğru bir seçimdir.
