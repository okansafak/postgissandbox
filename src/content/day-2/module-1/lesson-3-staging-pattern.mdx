## Öğrenim Hedefleri

Ham veriyi doğrudan hedef tabloya yazmak, temizlenmemiş verilerin kalıcı olarak sisteme girmesine yol açar. Staging deseni bu riski yönetmenin endüstri standardı yaklaşımıdır: ham veri önce ayrı bir tabloda saklanır, doğrulanır, ardından temizlenmiş haliyle hedef tabloya aktarılır.

- Staging tablosu desenini açıklamak
- Ham veriyi temizleyip hedef tabloya aktarmak
- `ON CONFLICT`, `WHERE` koşulları ve NULL kontrollerini uygulamak

---

## Staging Deseni

Ham veriyi doğrudan hedef tabloya koymak hatalara zemin hazırlar. Staging deseni üç aşamalı bir akış önerir:

```
Ham veri → Staging (TEXT) → Temizle/Doğrula → Hedef (geometry)
```

Bu yaklaşımın avantajları:
- Ham veri kaybolmaz (rollback imkânı)
- Dönüşüm hataları orijinali etkilemez
- Kaç satırın başarılı/başarısız olduğu raporlanabilir

---

## Uygulama Örneği

### Staging'e ham veri yükleme

<RunnableBlock
  sql={`-- 1. Staging tablosu — her şey TEXT
CREATE TABLE IF NOT EXISTS stg_mekanlar (
  id         SERIAL,
  ham_ad     TEXT,
  ham_tur    TEXT,
  ham_lon    TEXT,
  ham_lat    TEXT,
  yukleme_ts TIMESTAMPTZ DEFAULT NOW()
);

-- 2. Ham veri ekle (gerçekte dosyadan gelir)
INSERT INTO stg_mekanlar (ham_ad, ham_tur, ham_lon, ham_lat) VALUES
  ('Sultanahmet Camii', 'cami',    '28.9760',  '41.0055'),
  ('Aya Sofya',         'müze',    '28.9798',  '41.0082'),
  ('Kapalıçarşı',       'çarşı',   '28,9670',  '41.0106'),  -- virgüllü ondalık!
  ('Boğaz Köprüsü',     'köprü',   'hatalı',   '41.0463'),  -- geçersiz lon!
  ('Dolmabahçe',        'saray',   '29.0012',  NULL);        -- lat yok!

SELECT * FROM stg_mekanlar;`}
  label="Staging'e ham veri yükle"
  description="Tüm sütunları TEXT türünde olan bir staging tablosu oluşturulur; böylece geçersiz koordinatlar bile hata fırlатmadan kabul edilir. Beş satırdan üçü sorunlu: virgüllü ondalık, geçersiz metin ve NULL değer. yukleme_ts sütunu yükleme zamanını otomatik kaydeder ve sonraki aşamalarda hangi verinin ne zaman geldiğini izlemeye olanak tanır."
/>

### Doğrulama raporu

<RunnableBlock
  sql={`SELECT
  id,
  ham_ad,
  CASE
    WHEN ham_lon IS NULL OR ham_lat IS NULL THEN 'NULL koordinat'
    WHEN REPLACE(ham_lon,',','.') !~ '^-?[0-9]+\.?[0-9]*$' THEN 'Geçersiz lon'
    WHEN REPLACE(ham_lat,',','.') !~ '^-?[0-9]+\.?[0-9]*$' THEN 'Geçersiz lat'
    ELSE 'Geçerli'
  END AS durum
FROM stg_mekanlar;`}
  label="Staging doğrulama raporu"
  description="CASE ifadesi her satırı dört olası duruma göre sınıflandırır: NULL koordinat, geçersiz boylam, geçersiz enlem veya geçerli. REPLACE(ham_lon,',','.') virgüllü ondalıkları normalleştirir; regex ~'^-?[0-9]+\.?[0-9]*$' sayısal formata uygunluğu test eder. Bu rapor hedef tabloya aktarılmadan önce veri kalitesini sayısal olarak ortaya koyar."
/>

### Temiz satırları hedef tabloya aktarma

<RunnableBlock
  sql={`CREATE TABLE IF NOT EXISTS mekanlar (
  id   SERIAL PRIMARY KEY,
  ad   TEXT NOT NULL,
  tur  TEXT,
  geom GEOMETRY(POINT, 4326)
);

INSERT INTO mekanlar (ad, tur, geom)
SELECT
  ham_ad,
  ham_tur,
  ST_SetSRID(
    ST_MakePoint(
      REPLACE(ham_lon, ',', '.')::FLOAT,
      REPLACE(ham_lat, ',', '.')::FLOAT
    ),
    4326
  )
FROM stg_mekanlar
WHERE
  ham_lon IS NOT NULL AND ham_lat IS NOT NULL
  AND REPLACE(ham_lon,',','.') ~ '^-?[0-9]+\.?[0-9]*$'
  AND REPLACE(ham_lat,',','.') ~ '^-?[0-9]+\.?[0-9]*$'
ON CONFLICT DO NOTHING;

-- Kaç satır aktarıldı?
SELECT COUNT(*) AS hedef_kayit FROM mekanlar;`}
  label="Temiz satırları aktar"
  description="WHERE koşulu yalnızca doğrulama raporunda 'Geçerli' olarak işaretlenen satırları seçer: NULL olmayan ve sayısal regex'e uyan koordinatlar. REPLACE virgüllü ondalıkları ::FLOAT cast öncesinde normalleştirir. ON CONFLICT DO NOTHING aynı sorgunun tekrar çalışmasında çakışmaları önler. Sonuç sayısının 2 (geçerli satır sayısı) olması beklenir."
/>

## Yaygın Hatalar

1. **`::FLOAT` geçersiz metinde hata fırlatır.** `'hatalı'::FLOAT` → `ERROR`. Regex koşuluyla önceden filtreleme yapmak zorunludur.
2. **Staging tablosunu `DROP` ile silme.** Yükleme tamamlandıktan sonra staging tablosunu `TRUNCATE` ile temizlemek yeterlidir — şema korunur, veri gider. `DROP` şemayı da siler ve sonraki yükleme döngüsünde yeniden oluşturma gerekir.
3. **Coğrafi duplicate tespiti.** `ON CONFLICT DO NOTHING` yalnızca PRIMARY KEY/UNIQUE kısıt çakışmasında atlar. Coğrafi olarak aynı konumdaki tekrar kayıtları filtrelemek için `WHERE NOT EXISTS (SELECT 1 FROM mekanlar WHERE ST_DWithin(...))` kalıbı kullanılmalıdır.
