## Öğrenim Hedefleri

GiST her durum için en verimli seçenek değildir. GPS takip verisi veya POI tabloları gibi yoğun nokta kümelerinde SP-GIST alternatif bir performans profili sunar. Bu ders, iki indeks türünün yapısal farklarını ve hangi senaryoda hangisinin tercih edileceğini ele almaktadır.

- SP-GIST ile GiST arasındaki yapısal farkı açıklamak
- SP-GIST'in avantajlı olduğu durumları belirlemek
- `CREATE INDEX USING spgist` sözdizimini yazmak

---

## SP-GIST (Space-Partitioned GIST)

| Özellik | GiST | SP-GIST |
|---|---|---|
| Yapı | R-Tree (örtüşen bbox'lar) | Quadtree / K-D tree (bölünmüş uzay) |
| Örtüşme | Evet | Hayır (bölgeler ayrı) |
| En iyi tablo | Karma geometri, poligon | Nokta yoğun, uniform dağılım |
| İndeks boyutu | Daha büyük | Daha küçük |
| Nokta sorgusu | İyi | Çok iyi |
| Büyük poligon | İyi | Zayıf |

**Pratik kural:** GPS takip verisi veya POI tablosu → SP-GIST dene. Poligon tabloları → GiST.

## Uygulama Örneği

### SP-GIST indeks ile GPS sorgusu

<RunnableBlock
  sql={`CREATE TABLE IF NOT EXISTS gps_izler (
  id   SERIAL PRIMARY KEY,
  arac TEXT,
  zaman TIMESTAMPTZ,
  geom GEOMETRY(POINT, 4326)
);

INSERT INTO gps_izler (arac, zaman, geom)
SELECT
  'Araç-' || (1 + i % 10),
  NOW() - (i || ' minutes')::interval,
  ST_SetSRID(ST_MakePoint(28.9 + random()*0.5, 40.9 + random()*0.3), 4326)
FROM generate_series(1, 2000) i
ON CONFLICT DO NOTHING;

-- SP-GIST indeks
CREATE INDEX IF NOT EXISTS idx_gps_spgist
  ON gps_izler USING spgist(geom);

ANALYZE gps_izler;

-- Sorgu planı
EXPLAIN ANALYZE SELECT COUNT(*) FROM gps_izler
WHERE ST_DWithin(
  geom::geography,
  ST_SetSRID(ST_MakePoint(29.1, 41.0),4326)::geography,
  20000);`}
  label="SP-GIST indeks ile GPS sorgusu"
  description="2000 araç GPS kaydı oluşturulur ve SP-GIST indeks eklenir. EXPLAIN ANALYZE hem sorgu planını hem de gerçek çalışma istatistiklerini gösterir: 'Index Scan using idx_gps_spgist' satırı ve 'actual time' değerleri. İndeks yokken aynı sorgunun 'Seq Scan' ile çalışacağı ve tüm 2000 satırı tarayacağı düşünüldüğünde fark anlamlıdır."
/>

### GiST vs SP-GIST boyutu karşılaştırması

<RunnableBlock
  sql={`-- GIST indeks de ekle (karşılaştırma için)
CREATE INDEX IF NOT EXISTS idx_gps_gist
  ON gps_izler USING gist(geom);

-- İki indeks mevcut
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS boyut
FROM pg_stat_user_indexes
WHERE tablename = 'gps_izler';`}
  label="GiST vs SP-GIST boyutu"
  description="Aynı tablo üzerinde hem GiST hem SP-GIST indeksi oluşturulur; pg_size_pretty her indeksin disk boyutunu okunabilir biçimde gösterir. Nokta yoğun ve uniform dağılımlı GPS verisi için SP-GIST genellikle daha küçük boyut üretir. Gerçek boyut farkı veri dağılımına bağlıdır."
/>

## Yaygın Hatalar

1. **SP-GIST tüm geometry tipleri için geçerli değildir.** PostGIS'te SP-GIST yalnızca `point` ve `box2d` tipleri için desteklenir. Polygon veya LineString sütunları için GiST kullanılmalıdır.
2. **Uniform dağılım yoksa GiST daha stabil çalışır.** Tüm noktalar tek bir bölgede kümelendiğinde SP-GIST quadtree dengesi bozulur ve GiST daha iyi performans verir.
3. **BRIN indeks zaman serisi için alternatiftir.** Çok büyük zaman serisi tablolarda `BRIN` indeksi çok küçük boyutuyla kullanılabilir; ancak mekansal sorgularda GiST/SP-GIST kadar seçici değildir.
