## PostgreSQL Veri Tipleri

**Hedef:** Tablolarda kullanılan tüm temel PostgreSQL veri tiplerini tanımak ve her tip için doğru kullanım senaryosunu belirlemek.

---

## Öğrenim Hedefleri

- Tam sayı, ondalık ve kayan noktalı tiplerin farkını açıklamak
- `NUMERIC` ile `REAL` arasındaki hassasiyet farkını bilmek
- `TIMESTAMPTZ` ile `TIMESTAMP` arasındaki kritik saat dilimi ayrımını anlamak
- `JSONB`, dizi, enum ve range tiplerini doğru bağlamda kullanmak

## Metin Tipleri

<RunnableBlock
  sql={`-- TEXT, VARCHAR ve CHAR davranış farkları
DROP TABLE IF EXISTS metin_karsilastir;
CREATE TABLE metin_karsilastir (
  alan_kodu   CHAR(6),
  proje_adi   VARCHAR(50),
  aciklama    TEXT
);
INSERT INTO metin_karsilastir VALUES
  ('PRJ001', 'Kentsel Dönüşüm', 'Kadıköy ilçesi parsel revizyonu'),
  ('PRJ002', 'Yol Ağı',         NULL);

SELECT
  alan_kodu,
  length(alan_kodu)   AS char_uzunluk,   -- her zaman 6
  octet_length(alan_kodu) AS char_bayt,
  proje_adi,
  length(proje_adi)   AS varchar_uzunluk,
  aciklama
FROM metin_karsilastir;`}
  label="Metin tipleri karşılaştırması"
  description="CHAR(6) değeri sağdan boşlukla doldurur — length() her zaman 6 döndürür; bu WHERE koşullarında beklenmedik eşleşmelere yol açabilir. VARCHAR(n) en fazla n karakter kabul eder, kullanılmayan alanı saklamaz. TEXT PostgreSQL'e özgüdür, uzunluk sınırı yoktur ve depolama/performans açısından VARCHAR ile eşdeğerdir — genel kullanım için TEXT tercih edilir. NULL boş string ile aynı değildir: length(NULL) → NULL."

## Tarih ve Saat Tipleri

<RunnableBlock
  sql={`-- Tarih ve saat tiplerini karşılaştır
SELECT
  '2024-06-15'::DATE                          AS tarih,
  '14:30:00'::TIME                            AS saat,
  '2024-06-15 14:30:00'::TIMESTAMP            AS timestamp_no_tz,
  '2024-06-15 14:30:00+03'::TIMESTAMPTZ       AS timestamp_tz,
  '2 years 3 months 10 days'::INTERVAL        AS aralik,
  CURRENT_DATE                                AS bugun,
  NOW()                                       AS simdi;`}
  label="Tarih ve saat tipleri"
  description="DATE yalnızca yıl-ay-gün bilgisi tutar; saat yoktur. TIME hh:mm:ss formatında saat saklar. TIMESTAMP tarih+saat tutar ancak saat dilimi bilgisi saklamaz — sunucu saat dilimi değiştiğinde yanlış yorumlanır. TIMESTAMPTZ ise değeri UTC'ye çevirerek saklar ve sorgu sırasında istemcinin saat dilimine göre gösterir — dağıtık sistemlerde zorunludur. INTERVAL iki tarih arasındaki fark veya belirli bir süreyi temsil eder."
/>

<RunnableBlock
  sql={`-- TIMESTAMPTZ aritmetiği ve INTERVAL
DROP TABLE IF EXISTS toplanti;
CREATE TABLE toplanti (
  id         SERIAL PRIMARY KEY,
  baslik     TEXT,
  baslangic  TIMESTAMPTZ NOT NULL,
  sure       INTERVAL    NOT NULL
);
INSERT INTO toplanti (baslik, baslangic, sure) VALUES
  ('Proje Kick-off',   '2024-09-01 09:00:00+03', '2 hours'),
  ('Haftalık Kontrol', '2024-09-08 10:00:00+03', '45 minutes'),
  ('Yıllık Değerlendirme', '2024-12-31 14:00:00+03', '3 hours 30 minutes');

SELECT
  baslik,
  baslangic,
  baslangic + sure                                 AS bitis_zamani,
  EXTRACT(DOW  FROM baslangic)                     AS hafta_gunu,  -- 0=Paz
  DATE_TRUNC('month', baslangic)::DATE             AS ay_baslangici,
  AGE(baslangic, CURRENT_TIMESTAMP)                AS ne_kadar_once
FROM toplanti
ORDER BY baslangic;`}
  label="TIMESTAMPTZ ile tarih aritmetiği"
  description="INTERVAL aritmetiği ay sonu ve gün sonu geçişlerini otomatik yönetir. EXTRACT(DOW ...) haftanın gününü sayısal (0=Pazar) döndürür. DATE_TRUNC zaman damgasını belirtilen birime yuvarlar; '2024-09-08 10:00' için DATE_TRUNC('month') → '2024-09-01'. AGE() iki zaman arasındaki farkı yıl-ay-gün-saat hassasiyetinde INTERVAL olarak döndürür."
/>

## Boolean ve İkili Veri

<RunnableBlock
  sql={`-- BOOLEAN ve üç değerli mantık (TRUE / FALSE / NULL)
DROP TABLE IF EXISTS proje_durumu;
CREATE TABLE proje_durumu (
  proje_adi   TEXT,
  tamamlandi  BOOLEAN,
  onaylandi   BOOLEAN
);
INSERT INTO proje_durumu VALUES
  ('Kadastro Revizyonu', TRUE,  TRUE),
  ('Yol Etüdü',          FALSE, TRUE),
  ('Çevre Planı',        TRUE,  NULL),  -- onay bekliyor
  ('Arşiv',              NULL,  FALSE); -- durum belirsiz

SELECT
  proje_adi,
  tamamlandi,
  onaylandi,
  tamamlandi AND onaylandi   AS her_ikisi,
  tamamlandi OR  onaylandi   AS en_az_biri,
  NOT tamamlandi             AS tamamlanmadi
FROM proje_durumu;`}
  label="Boolean üç değerli mantık"
  description="BOOLEAN TRUE, FALSE veya NULL (bilinmiyor) değerlerini alır. NULL içeren AND/OR işlemleri SQL'in üç değerli mantığına göre çalışır: TRUE AND NULL → NULL, FALSE AND NULL → FALSE, TRUE OR NULL → TRUE. NOT NULL → NULL. Bu davranış WHERE koşullarında beklenmedik satır kayıplarına yol açabilir; NULL değerleri için IS NULL / IS NOT NULL kullanılmalıdır."
/>

## JSON Tipleri

<RunnableBlock
  sql={`-- JSON vs JSONB karşılaştırması
DROP TABLE IF EXISTS mekansal_meta;
CREATE TABLE mekansal_meta (
  id       SERIAL PRIMARY KEY,
  katman   TEXT,
  meta_j   JSON,   -- metin saklama
  meta_jb  JSONB   -- ikili saklama, indekslenebilir
);
INSERT INTO mekansal_meta (katman, meta_j, meta_jb) VALUES
  ('Bina',    '{"kat": 5, "yil": 2018, "malzeme": "beton"}',
              '{"kat": 5, "yil": 2018, "malzeme": "beton"}'),
  ('Parsel',  '{"ada": "120", "parsel": "4", "alan_m2": 850.5}',
              '{"ada": "120", "parsel": "4", "alan_m2": 850.5}');

SELECT
  katman,
  meta_jb->>'malzeme'           AS malzeme,
  (meta_jb->>'kat')::INTEGER    AS kat_sayisi,
  (meta_jb->>'alan_m2')::NUMERIC AS alan,
  jsonb_object_keys(meta_jb)    AS anahtarlar
FROM mekansal_meta;`}
  label="JSONB sorgulama"
  description="JSON metni olduğu gibi saklar ve her sorguda ayrıştırır. JSONB ikili formatta saklar: sıralama değişebilir, yinelenen anahtarlar kaldırılır, ancak sorgular çok daha hızlıdır ve GIN indeks ile desteklenir. ->> operatörü JSON'dan metin değeri çeker; sayısal işlem için ::INTEGER veya ::NUMERIC cast'i gerekir. jsonb_object_keys() tüm anahtar adlarını satır olarak döndürür. Genel kullanımda JSONB tercih edilmelidir."
/>

<RunnableBlock
  sql={`-- JSONB GIN indeksi ve @> operatörü
DROP TABLE IF EXISTS ozellikler;
CREATE TABLE ozellikler (
  id   SERIAL PRIMARY KEY,
  veri JSONB
);
INSERT INTO ozellikler (veri) VALUES
  ('{"tip": "okul", "ilce": "Kadıköy", "kapasite": 500}'),
  ('{"tip": "hastane", "ilce": "Şişli", "yatak": 200}'),
  ('{"tip": "okul", "ilce": "Beşiktaş", "kapasite": 300}'),
  ('{"tip": "park", "ilce": "Kadıköy", "alan_m2": 12000}');

-- @> operatörü: JSONB içinde belirtilen alt belgeyi içeriyor mu?
SELECT veri->>'tip' AS tip, veri->>'ilce' AS ilce
FROM ozellikler
WHERE veri @> '{"tip": "okul"}';`}
  label="JSONB içerik sorgusu (@> operatörü)"
  description="@> operatörü 'sol JSONB, sağ JSONB'yi içeriyor mu?' sorusunu test eder. GIN indeks oluşturulduğunda (CREATE INDEX ON tablo USING GIN(veri)) bu operatör tam tarama yerine indeks kullanır. Benzer şekilde ? operatörü 'bu anahtar mevcut mu?' sorgular. JSONB esnekliği sayesinde şemanın önceden bilinmediği coğrafi nesne özelliklerini saklamak için PostGIS projelerinde yaygın kullanılır."

## Dizi (Array) Tipleri

<RunnableBlock
  sql={`-- Dizi tipleri: INTEGER[], TEXT[], NUMERIC[]
DROP TABLE IF EXISTS izleme_istasyonu;
CREATE TABLE izleme_istasyonu (
  id            SERIAL PRIMARY KEY,
  ad            TEXT,
  koordinat     NUMERIC[],   -- [lon, lat]
  olcum_saatleri INTEGER[],  -- gün içi saat dilimi (0-23)
  etiketler     TEXT[]
);
INSERT INTO izleme_istasyonu (ad, koordinat, olcum_saatleri, etiketler) VALUES
  ('Kadıköy İstasyonu', ARRAY[29.023, 40.990], ARRAY[6,12,18], ARRAY['hava','gurultu']),
  ('Şişli İstasyonu',   ARRAY[28.988, 41.060], ARRAY[8,14,20], ARRAY['hava','trafik']);

SELECT
  ad,
  koordinat[1]               AS boylam,
  koordinat[2]               AS enlem,
  array_length(etiketler, 1) AS etiket_sayisi,
  etiketler[1]               AS ilk_etiket,
  'hava' = ANY(etiketler)    AS hava_var_mi
FROM izleme_istasyonu;`}
  label="Dizi tipi kullanımı"
  description="PostgreSQL dizileri 1'den başlar (0'dan değil). array_length(dizi, boyut) dizinin uzunluğunu döndürür. ANY(dizi) bir değerin dizi içinde mevcut olup olmadığını test eder. UNNEST(dizi) diziyi satırlara açar — GROUP BY veya JOIN işlemlerinde kullanılabilir. Dizi tipleri normalleştirilmiş ilişkisel tasarıma göre ikinci dereceden tercih edilir; ancak sabit veya küçük boyutlu listeler için pratiktir."
/>

DROP TABLE IF EXISTS ag_cihazi;
CREATE TABLE ag_cihazi (
  id       SERIAL PRIMARY KEY,
  cihaz    TEXT,
  ip_adr   INET,

);
INSERT INTO ag_cihazi (cihaz, ip_adr, ag_blok, mac_adr) VALUES

<RunnableBlock

## Range Tipleri

<RunnableBlock
  sql={`-- Range tipleri: int4range, daterange, tstzrange

## Pratik Uygulama: Kapsamlı Tablo Tasarımı

<RunnableBlock
  sql={`-- Tüm öğrenilen tipleri kullanan gerçekçi bir tablo
DROP TABLE IF EXISTS mekansal_proje;
CREATE TABLE mekansal_proje (
  id             UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  proje_kodu     CHAR(8)        NOT NULL UNIQUE,
  proje_adi      TEXT           NOT NULL,
  proje_tipi     TEXT           NOT NULL
                   CHECK (proje_tipi IN ('kentsel','altyapi','cevre','ulasim')),
  baslangic      DATE           NOT NULL,
  tamamlanma     TIMESTAMPTZ,
  butce_tl       NUMERIC(15, 2) DEFAULT 0 CHECK (butce_tl >= 0),
  tamamlandi_mi  BOOLEAN        DEFAULT FALSE,
  koordinatlar   NUMERIC[],
  etiketler      TEXT[],
  meta           JSONB,
  basvuru_ip     INET
);

INSERT INTO mekansal_proje
  (proje_kodu, proje_adi, proje_tipi, baslangic, butce_tl,
   koordinatlar, etiketler, meta, basvuru_ip)
VALUES
  ('PRJ00001', 'İstanbul Yaya Ağı', 'kentsel',   '2024-01-15', 2500000.00,
   ARRAY[28.979, 41.015], ARRAY['yaya','imar'],
   '{"faz": 1, "sorumlu": "CBS Birimi"}'::JSONB, '192.168.1.10'),
  ('PRJ00002', 'Ankara Park Analizi', 'cevre',   '2024-03-01', 850000.50,
   ARRAY[32.854, 39.930], ARRAY['yesil','analiz'],
   '{"faz": 2, "sorumlu": "Planlama"}'::JSONB,   '10.0.0.22');

SELECT
  proje_kodu,
  proje_adi,
  proje_tipi,
  baslangic,
  butce_tl,
  tamamlandi_mi,
  etiketler[1]                              AS ilk_etiket,
  meta->>'sorumlu'                          AS sorumlu_birim,
  koordinatlar[1]                           AS boylam,
  koordinatlar[2]                           AS enlem,
  (CURRENT_DATE - baslangic) || ' gün'      AS gecen_sure
FROM mekansal_proje
ORDER BY baslangic;`}
  label="Kapsamlı tablo tasarımı"
  description="DROP TABLE IF EXISTS önce eski versiyonu temizler; her çalıştırmada taze şema oluşur. Bu tablo Bölüm 0'da öğrenilen tüm tipleri bir arada kullanır: UUID (PRIMARY KEY), CHAR(8), TEXT + CHECK (kısıtlı tip alanı), DATE ve TIMESTAMPTZ, NUMERIC(15,2), BOOLEAN, NUMERIC[] (koordinat dizisi), TEXT[] (etiket dizisi), JSONB, INET. Süre hesabı DATE - DATE → INTEGER (gün sayısı) döndürür; bunu INTERVAL'e çevirmek için * INTERVAL '1 day' kullanılır. Bir sonraki adım: koordinatlar sütununun PostGIS geometry tipine dönüştürülmesi — bu Modül 1'in konusudur."
/>
