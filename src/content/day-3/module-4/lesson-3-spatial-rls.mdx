## Öğrenim Hedefleri

- `ST_Within` ile mekansal RLS politikası yazmak
- Kullanıcı yetki alanı poligonu bazlı erişim kısıtlaması uygulamak
- Mekansal politikanın performans etkisini değerlendirmek

---

## Mekansal Erişim Kontrolü

"İstanbul sorumlusu sadece İstanbul içindeki parselleri görsün."

```sql
CREATE POLICY mekansal_sinir ON parseller
  FOR SELECT
  USING (
    ST_Within(
      geom,
      (SELECT bolge FROM kullanici_bolgeleri WHERE kullanici = current_user)
    )
  );
```

### Mekansal RLS politikası

<RunnableBlock
  sql={`ALTER TABLE yetkili_noktalar ENABLE ROW LEVEL SECURITY;

-- Mekansal politika: kullanıcının yetki alanı içindeki noktalar
CREATE POLICY mekansal_erisim ON yetkili_noktalar
  FOR SELECT
  USING (
    ST_Within(
      geom,
      COALESCE(
        (SELECT bolge FROM kullanici_bolge WHERE kullanici = current_user),
        ST_MakeEnvelope(-180, -90, 180, 90, 4326)  -- superuser için tüm dünya
      )
    )
  );

-- Superuser her şeyi görür (politika bypass)
SELECT ad, ST_AsText(geom) AS konum FROM yetkili_noktalar;`}
  label="Mekansal RLS politikası"
  description="CREATE POLICY USING bloğu her satır için ST_Within kontrolü uygular. COALESCE kullanıcının kayıtlı yetki alanını çeker; kullanıcı kullanici_bolge tablosunda yoksa tüm dünyayı kapsayan fallback bbox kullanılır. Superuser BYPASSRLS ile politikayı atlar ve tüm 4 satırı görür. Normal kullanıcı yalnızca kendi bölge poligonu içindeki noktaları görür."
/>

### İstanbul kullanıcısı simülasyonu

<RunnableBlock
  sql={`-- Politika aktif ama superuser — manuel filtre ile test
-- Gerçek sistemde: SET ROLE istanbul_usr; ile test edilir
SELECT
  n.ad,
  CASE WHEN ST_Within(n.geom, b.bolge) THEN 'Erişilebilir' ELSE 'Kısıtlı'
  END AS erisim
FROM yetkili_noktalar n
CROSS JOIN kullanici_bolge b
WHERE b.kullanici = 'istanbul_usr';`}
  label="İstanbul kullanıcısı simülasyonu"
  description="Superuser oturumunda SET ROLE ile kullanıcı değiştirilemediğinden mekansal politika manuel simüle edilir. CROSS JOIN kullanıcının bölge poligonunu her noktayla eşleştirir. ST_Within İstanbul bbox içindeki noktalar için 'Erişilebilir', dışındakiler için 'Kısıtlı' döndürür. Sonuçta 2 İstanbul noktası erişilebilir, Ankara ve Orta Anadolu kısıtlı görünür."

## Pratik Uygulama

<RunnableBlock
  sql={`-- Ankara kullanıcısının görebileceği noktalar
SELECT
  n.ad,
  ROUND(ST_Distance(n.geom::geography, ST_Centroid(b.bolge)::geography)::numeric / 1000, 1) AS merkeze_km
FROM yetkili_noktalar n
JOIN kullanici_bolge b ON b.kullanici = 'ankara_usr'
WHERE ST_Within(n.geom, b.bolge);`}
  label="Ankara kullanıcısı görünür noktalar"
  description="WHERE ST_Within(n.geom, b.bolge) Ankara bölgesi içindeki noktaları seçer. ST_Distance her noktanın Ankara bölgesinin merkezine olan uzaklığını km cinsinden hesaplar. Yalnızca 'Ankara Noktası 1' eşleşmeli; İstanbul ve Orta Anadolu noktaları filtrelenmelidir."

## Yaygın Hatalar

1. **Mekansal politika GIST indeksini doğrudan kullanmaz:** `ST_Within(geom, subquery)` her satırda alt sorguyu çalıştırır. Performans için yetki alanı oturumda önbelleğe alınmalı veya materialized view kullanılmalıdır.
2. **`current_user` vs `session_user`:** Rol değiştirildiğinde `current_user` değişir, `session_user` değişmez. Politikada `current_user` güvenli seçimdir.
3. **NULL geometri:** Kullanıcının yetki alanı yoksa `COALESCE` ile fallback verilmezse politika tüm satırları reddeder.
