## Capstone Bölüm 4: Üretim Ortamı ve İş Metrikleri

**Hedef:** Yüksek erişilebilirlik mimarisini, bağlantı havuzlamayı ve iş sonuçlarını değerlendiren metrikleri incelemek.

---

## Öğrenim Hedefleri

- Primary-replica-backup üç katmanlı HA mimarisini kavramak
- pgBouncer transaction mode ile bağlantı yönetimini anlamak
- İzleme ve uyarı stratejisini yapılandırmak
- Teknik yatırımı iş etkisiyle ilişkilendirmek

---

## Üretim Mimarisi

| Bileşen | Yapılandırma |
|---|---|
| Veritabanı | PostgreSQL 15 + PostGIS 3.3 |
| Replikasyon | Primary + Replica + Yedek Replica |
| Bağlantı havuzu | pgBouncer (transaction mode) |
| İzleme | Prometheus + Grafana |
| Failover süresi | 30 saniye altı |

---

## Uygulama Örneği

### Replikasyon düğüm yapısı

<RunnableBlock
  sql={`-- Replikasyon düğüm yapısı
SELECT 
  'Primary' AS rol,
  '16 core' AS cpu,
  '256 GB' AS ram,
  'Istanbul-DC1' AS konum,
  'Aktif' AS durum
UNION ALL
SELECT 'Replica', '12 core', '128 GB', 'Istanbul-DC2', 'Standby'
UNION ALL
SELECT 'Yedek Replica', '8 core', '64 GB', 'Yedek-DC', 'Reserve';

-- Bağlantı havuzu yapılandırması
SELECT 
  'pgBouncer' AS havuzlayici,
  'Transaction mode' AS havuz_modu,
  1000 AS max_istemci,
  50 AS varsayilan_havuz_boyutu,
  '%97+ hit oranı' AS hedef,
  'CPU yükü %99 azalma' AS kazanim;`}
  label="Veritabanı HA ve bağlantı havuzlama"
  description="Üç katmanlı replikasyon yapısı coğrafi yedeklilik sağlar: primary aktif yazma düğümü, replica okuma offloading ve otomatik failover hedefi, yedek replica ise felaket kurtarma için ayrı veri merkezinde bulunur. Transaction mode pgBouncer her SQL bloğundan sonra bağlantıyı havuza iade eder; bu mod PostGIS uzun işlemleri için dikkatli değerlendirilmelidir. %97 hit oranı her 100 istekten 97'sinin yeni TCP bağlantısı açmadan karşılandığı anlamına gelir."
/>

### İzleme ve uyarı yapılandırması

<RunnableBlock
  sql={`-- Temel izleme metrikleri ve eşik değerleri
SELECT 'Metrik' AS metrik, 'Mevcut' AS mevcut, 'Eşik' AS esik, 'Durum' AS durum
UNION ALL
SELECT 'Replikasyon gecikmesi (MB)', '45', '< 100', 'OK'
UNION ALL
SELECT 'Sorgu gecikmesi p95 (ms)', '280', '< 500', 'OK'
UNION ALL
SELECT 'Disk kullanımı (%)', '65', '< 85', 'OK'
UNION ALL
SELECT 'CPU kullanımı (%)', '48', '< 80', 'OK'
UNION ALL
SELECT 'Havuz hit oranı (%)', '97', '> 95', 'OK';

-- Uyarı yönlendirme
SELECT 'Seviye' AS seviye, 'Hedef' AS hedef, 'Yanıt Süresi' AS yanit
UNION ALL
SELECT 'Kritik', 'SMS + PagerDuty', '5 dakika altı'
UNION ALL
SELECT 'Uyarı', 'Slack', '15 dakika altı'
UNION ALL
SELECT 'Bilgi', 'Yalnızca günlük', 'Günlük inceleme';`}
  label="İzleme ve uyarı yapılandırması"
  description="Her metrik için mevcut değer ve eşik değeri karşılaştırılır; eşik aşımında uyarı kanalı tetiklenir. Replikasyon gecikmesi 100 MB'ı aştığında replica primary'den çok geride kalmış olup failover durumunda veri kaybı riski artar. Uyarı seviyeleri olayın ciddiyetine göre kanallanır: kritik olaylar gece de müdahale gerektiren SMS + PagerDuty'ye, uyarılar iş saatinde izlenen Slack'e yönlendirilir."
/>

## Pratik Uygulama

<RunnableBlock
  sql={`-- Platform iş metrikleri özeti
SELECT 'İş Metriği' AS metrik, 'Değer' AS deger, 'Hedef / Önceki' AS referans
UNION ALL
SELECT 'Ambulans yanıt süresi', '8 dakika', '< 8 dk (önceki: 15 dk)'
UNION ALL
SELECT 'Ortalama teslimat maliyeti', '32 TL', '32 TL (önceki: 50 TL)'
UNION ALL
SELECT 'Platform uptime', '%99.99', '%99.99 (önceki: %99.9)'
UNION ALL
SELECT 'Yatırım getirisi (ROI)', '%441', '> %100 hedefi'
UNION ALL
SELECT 'Geri ödeme süresi', '2.2 ay', 'Hedef: < 6 ay';`}
  label="İş başarı metrikleri"
  description="Teknik yatırımın iş sonuçlarına dönüşümünü gösteren metrikler: ambulans yanıt süresi operasyonel iyileşmeyi, teslimat maliyeti lojistik optimizasyon kazanımını, uptime altyapı güvenilirliğini temsil eder. ROI = (net kazanç / yatırım maliyeti) * 100 formülüyle hesaplanır; %441 oran yatırımın 5.4 katına döndüğünü ifade eder. 2.2 aylık geri ödeme süresi yüksek öncelikli proje onayı için yönetim sunum materyalinde kullanılır."
