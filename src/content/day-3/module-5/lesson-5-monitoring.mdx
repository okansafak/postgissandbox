## Öğrenim Hedefleri

- `pg_stat_activity` ile aktif sorguları izlemek
- Buffer cache hit oranını hesaplamak
- `pg_stat_user_tables` ile dead tuple ve seq scan uyarılarını okumak

---

## PostgreSQL Sağlık Göstergeleri

| Gösterge | Hedef | Sorun işareti |
|---|---|---|
| Cache hit oranı | %99'dan yüksek | %95'in altı → shared_buffers artırılmalı |
| Dead tuple oranı | %5'in altı | %20'nin üzeri → autovacuum sıklığı artırılmalı |
| Seq scan oranı | %10'un altı (büyük tablolar) | %50'nin üzeri → indeks eksik |
| Uzun süreli sorgular | 1 dakikadan az | 5 dakikadan uzun → yavaş sorgu analizi |
| Bağlantı kullanımı | %80'in altı | %90'ın üzeri → PgBouncer değerlendirilmeli |

## Uygulama Örneği

### Aktif sorgular

<RunnableBlock
  sql={`-- Çalışan sorgular + süre
SELECT
  pid,
  usename AS kullanici,
  application_name AS uygulama,
  state,
  EXTRACT(EPOCH FROM (now() - query_start))::int AS sure_sn,
  LEFT(query, 80) AS sorgu
FROM pg_stat_activity
WHERE state != 'idle'
  AND pid != pg_backend_pid()
ORDER BY sure_sn DESC NULLS LAST;`}
  label="Aktif sorgular"
  description="pg_stat_activity tüm backend işlemlerini listeler. WHERE state != 'idle' yalnızca aktif veya beklemedeki bağlantıları gösterir. EXTRACT(EPOCH FROM (now() - query_start)) saniye cinsinden sorgu süresini hesaplar. ORDER BY sure_sn DESC en uzun süren sorguyu üste getirir. LEFT(query, 80) uzun sorguları kısaltır."
/>

### Cache hit oranı

<RunnableBlock
  sql={`-- Buffer cache hit oranı — veritabanı bazında
SELECT
  datname AS veritabani,
  blks_hit AS onbellek_hit,
  blks_read AS disk_okuma,
  ROUND(
    blks_hit::numeric / NULLIF(blks_hit + blks_read, 0) * 100, 2
  ) AS hit_oran_pct
FROM pg_stat_database
WHERE datname = current_database();`}
  label="Cache hit oranı"
  description="blks_hit shared_buffers'dan karşılanan okuma sayısını, blks_read diskten okunan blok sayısını gösterir. hit_oran_pct = blks_hit / (blks_hit + blks_read) * 100 formülüyle hesaplanır. NULLIF(blks_hit + blks_read, 0) sıfıra bölme hatasını önler. %99'un altındaki oran shared_buffers'ın artırılması gerektiğine işaret eder."
/>

### Tablo sağlık raporu

<RunnableBlock
  sql={`-- Kapsamlı tablo sağlık raporu
SELECT
  relname AS tablo,
  n_live_tup AS canli,
  n_dead_tup AS olu,
  ROUND(
    n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 1
  ) AS olu_oran_pct,
  seq_scan,
  idx_scan,
  ROUND(
    idx_scan::numeric / NULLIF(seq_scan + idx_scan, 0) * 100, 1
  ) AS idx_oran_pct,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
WHERE n_live_tup > 100
ORDER BY olu_oran_pct DESC NULLS LAST
LIMIT 10;`}
  label="Tablo sağlık raporu"
  description="olu_oran_pct yüksek tablolar VACUUM öncelik sıralamasında öne alınmalıdır. idx_oran_pct düşük tablolarda indeks eksik veya planlayıcı indeksi tercih etmiyor demektir. last_autovacuum çok eski ise autovacuum çalışmıyor olabilir; n_dead_tup yüksekse autovacuum_vacuum_scale_factor küçültülmelidir. WHERE n_live_tup > 100 boş veya çok küçük tabloları filtreler."
/>

### Uzun süreli sorgu tespiti

<RunnableBlock
  sql={`-- pg_stat_statements (extension gerektirir — olmayabilir)
-- Yoksa pg_stat_activity ile uzun süreli sorguları izle
SELECT
  pid,
  usename,
  EXTRACT(EPOCH FROM (now() - query_start))::int AS sure_sn,
  wait_event_type,
  wait_event,
  LEFT(query, 100) AS sorgu
FROM pg_stat_activity
WHERE state = 'active'
  AND query_start < now() - interval '5 seconds'
  AND pid != pg_backend_pid()
ORDER BY sure_sn DESC;`}
  label="Uzun süreli sorgu tespiti"
  description="5 saniyeden uzun çalışan aktif sorgular listelenir. wait_event_type ve wait_event beklemenin nedenini gösterir: 'Lock' kilit bekleme, 'IO' disk bekleme, 'Client' uygulama bekleme anlamına gelir. Üretimde bu sorgu pg_stat_statements ile birlikte kullanılmalıdır; pg_stat_statements tamamlanan sorguların istatistiğini de tutar."
/>

## Pratik Uygulama

<RunnableBlock
  sql={`-- Tek sorguda veritabanı özeti
SELECT
  (SELECT COUNT(*) FROM pg_stat_activity WHERE state = 'active') AS aktif_baglanti,
  (SELECT ROUND(blks_hit::numeric/NULLIF(blks_hit+blks_read,0)*100,1)
   FROM pg_stat_database WHERE datname = current_database()) AS cache_hit_pct,
  (SELECT SUM(n_dead_tup) FROM pg_stat_user_tables) AS toplam_olu_satir,
  (SELECT COUNT(*) FROM pg_stat_user_tables WHERE n_dead_tup > n_live_tup * 0.1) AS vakum_gereken,
  pg_size_pretty(pg_database_size(current_database())) AS db_boyutu;`}
  label="Veritabanı özet dashboardu"
  description="Beş alt sorgu tek satırda birleştirilir: aktif bağlantı sayısı, cache hit yüzdesi, tüm tablolardaki toplam ölü satır, %10'dan fazla ölü satır oranına sahip tablo sayısı ve veritabanı toplam boyutu. Bu özet sorgu periyodik sağlık kontrolü veya monitoring dashboard'una entegrasyon için kullanılır."
/>

## Yaygın Hatalar

1. **`pg_stat_statements` şart:** Yavaş sorgu analizi için en güçlü araçtır. `CREATE EXTENSION pg_stat_statements` ve `shared_preload_libraries = 'pg_stat_statements'` yapılandırması gerektirir.
2. **İzleme istatistikleri sıfırlanabilir:** `pg_stat_reset()` tüm istatistikleri temizler; sunucu yeniden başlatmasında da sıfırlanır.
3. **`pg_stat_activity` yeterli değil:** Tamamlanan sorgular gösterilmez. Kapsamlı analiz için log + `pg_stat_statements` kombinasyonu gerekir.
