## Öğrenim Hedefleri

- PgBouncer'ın işlevini ve neden gerekli olduğunu açıklamak
- Session, transaction ve statement pooling modlarını karşılaştırmak
- PostGIS için hangi pool modunun uygun olduğunu belirlemek

---

## Bağlantı Havuzu

**Problem:** Her PostgreSQL bağlantısı yaklaşık 5–10 MB RAM tüketir. 1000 eş zamanlı kullanıcı yalnızca bağlantılar için 5–10 GB RAM anlamına gelir.

**PgBouncer çözümü:**
- Uygulama → PgBouncer (binlerce bağlantı)
- PgBouncer → PostgreSQL (100–300 gerçek bağlantı)
- Havuz: boş bağlantıları bekleyen sorgulara dağıtır

---

## Pool Modları

| Mod | Bağlantı tutulur | PostGIS uyumu | Kullanım |
|---|---|---|---|
| **Session** | Oturum boyunca | Tam uyumlu | Prepared statements, SET |
| **Transaction** | İşlem boyunca | Çoğunlukla uyumlu | Web uygulaması |
| **Statement** | Her sorgu | Çok ifadeli SQL desteklenmez | Çok basit okuma |

**PostGIS için:** Transaction veya Session pooling tercih edilmeli. Statement pooling, `ST_*` fonksiyon çağrıları ve prepared statements nedeniyle sorun çıkarır.

## Uygulama Örneği

### Bağlantı durumu

<RunnableBlock
  sql={`-- Mevcut bağlantı sayısı
SELECT
  COUNT(*) AS toplam_baglanti,
  COUNT(*) FILTER (WHERE state = 'active') AS aktif,
  COUNT(*) FILTER (WHERE state = 'idle') AS bosta,
  MAX(EXTRACT(EPOCH FROM (now() - state_change))::int) AS en_uzun_bosta_sn
FROM pg_stat_activity
WHERE pid != pg_backend_pid();`}

### Bağlantı ve bellek ayarları

<RunnableBlock
  sql={`-- PostgreSQL bağlantı limitleri
SELECT
  name,
  setting,
  unit
FROM pg_settings
WHERE name IN (
  'max_connections',
  'superuser_reserved_connections',
  'work_mem',
  'shared_buffers'
)
ORDER BY name;`}
  label="Bağlantı ve bellek ayarları"

PgBouncer max_connections'ı PostgreSQL seviyesinde düşük tutarak belleği korur."

## PgBouncer Konfigürasyon Şablonu

```ini
# pgbouncer.ini
[databases]
cbs_db = host=127.0.0.1 port=5432 dbname=cbs_db

[pgbouncer]
listen_port = 6432
listen_addr = 127.0.0.1
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

pool_mode = transaction        # PostGIS için
max_client_conn = 1000         # Uygulama bağlantıları
default_pool_size = 100        # PostgreSQL bağlantıları

# PostGIS uyumluluk
ignore_startup_parameters = extra_float_digits
server_reset_query = DISCARD ALL
```

## Yaygın Hatalar

1. **Transaction pooling + prepared statements:** PgBouncer transaction modunda prepared statements farklı bağlantıya dağıtılabilir ve hata üretir. `prepare_threshold = 0` ile devre dışı bırakılmalı veya session pool kullanılmalıdır.
2. **PostGIS SET komutları:** `SET search_path`, `SET enable_seqscan` gibi komutlar transaction pooling'de oturumlar arası taşınmaz; her sorguda yeniden gönderilmelidir.
3. **PgBouncer izleme:** `SHOW POOLS`, `SHOW CLIENTS`, `SHOW SERVERS` komutları PgBouncer admin konsolundan çalıştırılır (port 6432, `pgbouncer` veritabanına bağlanarak).
