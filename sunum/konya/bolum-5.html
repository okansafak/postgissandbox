5.2.2 — Tasarım Kararları: EPSG, Z Koordinatı, Tip Kısıtlama
06 / 22
3 Kritik
Karar
📐 Karar 1: EPSG:5255 (TUREF / TM33)
Konya ~32.49°E boylamında. TUREF TM33 zonu (merkez meridyen 33°E) Konya için doğru kesit. Metre cinsinden → uzunluk/alan hesapları doğrudan metre çıkar, dönüşüm gerekmez.
⚠️ ST_Length WGS84'te derece, 5255'te metre döner.
📏 Karar 2: Z Koordinatı (Derinlik)
Altyapı hatları yeraltındadır — farklı derinliklerde çakışabilirler. LINESTRINGZ / POINTZ ile Z = derinlik (negatif = yeraltı).
Gaz -0.8m, fiber -0.3m: 2D'de üst üste görünür, ST_3DDistance ile 0.5m ayrım ölçülür.
🔒 Karar 3: geometry(Tip, SRID) ile Kısıtlama
geom geometry(LINESTRINGZ, 5255) → PostgreSQL yanlış tip veya SRID'li INSERT'i reddeder. Eski AddGeometryColumn() yaklaşımı artık kullanılmıyor.

🗺️ QGIS otomatik projeksiyon
Harita projesi EPSG:3857 bile olsa QGIS, PostGIS'ten 5255 sütununu okurken arka planda ST_Transform uygular — ek view oluşturmak zorunlu değil.
ST_3DDistance ne zaman kullanırım? ▼
İki hat 2D'de 0 mesafede (üst üste) ama farklı derinliklerde olabilir. ST_3DDistance(gas.geom, fiber.geom) gerçek uzaysal mesafeyi (örn. 0.5m) döner. Güvenlik koridoru ve çakışma analizinde kritik — sadece 2D baksaydın tüm hatlar çakışıyor görünürdü.

5.3.1 — DDL: Şema, Kurumlar Tablosu ve Mahalle
08 / 22
Şema
Oluşturma
Her proje kendi şemasına girmeli. public'e yazmak ekip çalışmasında karışıklık yaratır.
📦 CREATE SCHEMA aykome
Tüm AYKOME tablolarını yalıtır. Farklı şemalar (cbs, planlama) ile çakışma olmaz. SET search_path = aykome ile şema adı yazmadan erişilir.
🏢 utility_company — Referans Tablo
Normalleştirmenin kalbi. Kurum adı bir kez burada; hatlar ve izinler yalnızca kurum_id taşır. Kurum adı değişirse tek satır güncellenir, tüm bağlantılar otomatik güncellenir.
🏘️ mahalle — Coğrafi Referans
Konya'da 627 mahalle var. Kazı izinleri ve yol segmentleri bu tabloya FK ile bağlanır → mahalle bazlı istatistik ve rapor üretmek tek JOIN ile mümkün.
SQL — Şema ve İlk İki Tablo
-- ▶ 1) AYKOME şeması oluştur CREATE SCHEMA IF NOT EXISTS aykome; -- Oturum boyunca şema adı yazmadan erişim: -- SET search_path = aykome, public; -- ▶ 2) Altyapı kurumları referans tablosu CREATE TABLE aykome.utility_company ( id SERIAL PRIMARY KEY, ad TEXT NOT NULL, kisa_ad TEXT NOT NULL UNIQUE, -- 'KOSKI','GAZNET','MEDAS' tip TEXT NOT NULL, -- 'SU','KANAL','GAZ','ELEKTRIK','TELEKOM' renk TEXT DEFAULT '#888888' -- harita görünümü için hex renk ); COMMENT ON TABLE aykome.utility_company IS 'AYKOME üyesi altyapı kurumları'; COMMENT ON COLUMN aykome.utility_company.renk IS 'QGIS katman rengi (hex, örn. #1565C0)'; -- ▶ 3) Mahalle sınırları (coğrafi referans) CREATE TABLE aykome.mahalle ( id SERIAL PRIMARY KEY, ad TEXT NOT NULL, ilce TEXT NOT NULL, -- 'Selçuklu', 'Meram', 'Karatay' nufus INTEGER, geom geometry(POLYGON, 5255) ); CREATE INDEX idx_mahalle_geom ON aykome.mahalle USING GiST(geom); COMMENT ON TABLE aykome.mahalle IS 'Konya mahalle idari sınırları (EPSG:5255 TUREF/TM33)';
COMMENT ON ne işe yarar pratikte? ▼
\d+ aykome.utility_company veya pgAdmin'de tabloyu açtığında COMMENT görünür. QGIS DB Manager'da tooltip olarak çıkar. Farklı kurumların eriştiği ortak şemalarda ekip içi dokümantasyon için kritik — 6 ay sonra sütunun ne anlama geldiğini herkes okuyabilir.

5.3.2 — DDL: Altyapı Hatları, Rögarlar ve Yol Segmentleri
09 / 22
Hat ve
Nokta
Tüm hat geometrileri LINESTRINGZ, nokta geometrileri POINTZ — Z ekseni yeraltı derinliğini taşır.
🔵 utility_line — Tüm Altyapı Hatları
Su, kanal, gaz, elektrik, telekom tek tabloda — tip sütunuyla ayrılır. capi_mm boru çapı, basinc_bar gaz/su basıncı, derinlik_m yerden mesafe (negatif = yeraltı).
🟣 manhole — Rögar ve Vanalar
tip: rögar, baca, vana, trafo_odasi, pompa_istasyonu. Her bakım erişim noktası burada. Z koordinatı kapak kotunu (denizden yükseklik) veya derinliği taşır.
🟢 road_segment — Asfalt Garanti Takibi
son_asfalt_tarihi + garanti_bitisi → kazı izni garanti ihlali sorgularında kullanılır. mahalle_id FK ile mahalle bazlı yol envanteri üretilir.
SQL — Ana Geometri Tabloları
-- ▶ 4) Altyapı hatları (su, kanal, gaz, elektrik, telekom) CREATE TABLE aykome.utility_line ( id SERIAL PRIMARY KEY, kurum_id INTEGER NOT NULL REFERENCES aykome.utility_company(id), tip TEXT NOT NULL, -- 'su','kanal','dogalgaz','elektrik','fiber','data' capi_mm INTEGER, -- boru/kablo çapı mm derinlik_m NUMERIC(4,2), -- yerden derinlik (-0.80 = 80cm yeraltı) basinc_bar NUMERIC(5,2), -- gaz/su için basınç (bar) malzeme TEXT, -- 'PE','PVC','çelik','HDPE','betonarme' yapim_yili SMALLINT, durum TEXT DEFAULT 'aktif', -- 'aktif', 'pasif', 'hasar', 'yenileniyor' geom geometry(LINESTRINGZ, 5255) ); -- ▶ 5) Rögar, vana, baca, trafo noktaları CREATE TABLE aykome.manhole ( id SERIAL PRIMARY KEY, kurum_id INTEGER NOT NULL REFERENCES aykome.utility_company(id), tip TEXT NOT NULL, -- 'rögar','baca','vana','trafo_odasi','pompa' derinlik_m NUMERIC(4,2), yapim_yili SMALLINT, durum TEXT DEFAULT 'aktif', geom geometry(POINTZ, 5255) ); -- ▶ 6) Yol/cadde + asfalt garanti bilgisi CREATE TABLE aykome.road_segment ( id SERIAL PRIMARY KEY, ad TEXT NOT NULL, -- 'Mevlana Cad.', 'Alaaddin Blv.' mahalle_id INTEGER REFERENCES aykome.mahalle(id), kaplama TEXT DEFAULT 'asfalt', -- 'asfalt','parke','beton','stabilize' son_asfalt_tarihi DATE, garanti_bitisi DATE, -- genellikle son_asfalt + 5 yıl geom geometry(LINESTRING, 5255) );
Neden tip bazlı ayrı tablolar değil, tek tablo? ▼
Ayrı tablo alternatifi (su_hatti, kanal_hatti, gaz_hatti…) şema kısıtı sağlar ama 6+ tablo × N sütun = bakım yükü artar. Genel sorgularda UNION ALL zorunlu olur. Tek tablo + WHERE tip = 'dogalgaz' AYKOMEBİS'in gerçek yaklaşımına uyar ve B-Tree indeksi (idx_ul_tip) ile hızlıdır.

5.3.3 — DDL: Kazı İzin Tabloları ve İndeks Stratejisi
10 / 22
İzin +
İndeks
Kazı süreci iki tabloya ayrılır: başvuru (tek kayıt) + kurum görüşleri (N kayıt). İndeksler Bölüm 4'ten bilinen stratejiyle seçilir.
🟠 excavation_permit — Kazı Başvurusu
durum: beklemede → onaylandi → tamamlandi (ya da reddedildi). geom POLYGON QGIS'te sayısallaştırılarak çizilecek. basvuru_no UNIQUE resmi takip numarası.
⚫ permit_review — Kurum Görüşleri
Bir başvuruya her kurum (KOSKİ, GAZNET…) kendi görüşünü yazar. Tüm kurumlardan 'onay' gelince excavation_permit.durum güncellenebilir. reviewer_kurum_id FK hangi kurumun incelediğini tutar.
⚡ İndeks Seçimi (Bölüm 4 bağlantısı)
LINESTRING/POLYGON → GiST. Nokta → SP-GiST. Sadece 'beklemede' başvurular için Partial Index → indeks boyutu %90 küçülür.
SQL — İzin Tabloları + Tüm İndeks Bloğu
-- ▶ 7) Kazı ruhsat başvuruları CREATE TABLE aykome.excavation_permit ( id SERIAL PRIMARY KEY, basvuru_no TEXT UNIQUE, -- örn. 'AYK-2024-00142' basvuran_kurum_id INTEGER NOT NULL REFERENCES aykome.utility_company(id), mahalle_id INTEGER REFERENCES aykome.mahalle(id), basvuru_tarihi DATE NOT NULL DEFAULT CURRENT_DATE, durum TEXT NOT NULL DEFAULT 'beklemede', -- 'beklemede','onaylandi','reddedildi','tamamlandi' sebep TEXT NOT NULL, tahmini_sure_gun INTEGER, derinlik_m NUMERIC(4,2), geom geometry(POLYGON, 5255) ); -- ▶ 8) Kurum bazlı görüş / onay kayıtları CREATE TABLE aykome.permit_review ( id SERIAL PRIMARY KEY, permit_id INTEGER NOT NULL REFERENCES aykome.excavation_permit(id), reviewer_kurum_id INTEGER NOT NULL REFERENCES aykome.utility_company(id), gorus TEXT NOT NULL, -- 'onay', 'ret', 'kosullu_onay' aciklama TEXT, tarih TIMESTAMPTZ DEFAULT NOW() ); -- ═══════════════════════════════════════ -- İNDEKS STRATEJİSİ -- ═══════════════════════════════════════ -- Mekansal GiST (LINESTRING / POLYGON için doğru seçim) CREATE INDEX idx_ul_geom ON aykome.utility_line USING GiST(geom); CREATE INDEX idx_ep_geom ON aykome.excavation_permit USING GiST(geom); CREATE INDEX idx_rs_geom ON aykome.road_segment USING GiST(geom); CREATE INDEX idx_mh_geom ON aykome.mahalle USING GiST(geom); -- Mekansal SP-GiST (sadece POINT — quad-tree daha sığ ağaç) CREATE INDEX idx_mnh_geom ON aykome.manhole USING SPGIST(geom); -- B-Tree (kurum ve tip filtreleri) CREATE INDEX idx_ul_kurum ON aykome.utility_line(kurum_id); CREATE INDEX idx_ul_tip ON aykome.utility_line(tip); -- Partial Index (sadece bekleyen başvurular — %90 küçük) CREATE INDEX idx_ep_durum ON aykome.excavation_permit(durum) WHERE durum = 'beklemede';
Partial index neden %90 küçülür? ▼
Binlerce başvurunun çoğu 'tamamlandi' veya 'reddedildi' durumundadır — artık sorgulanmazlar. Partial index sadece 'beklemede' olanları (onlarca satır) indeksler. Sorguda WHERE durum = 'beklemede' şartı varsa PostgreSQL bu küçük indeksi seçer → tam tablo tarama yerine anında sonuç.
