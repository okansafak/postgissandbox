4.1.1 — GiST ve SP-GiST: Hangi Geometri Hangi İndeks?
02 / 23
GiST vs SP-GiST
Her geometri tipi için doğru indeks. İkisi de mekansaldır, ama çalışma mantıkları farklıdır.
📦 GiST (R-Tree) — İç İçe Klasörler
Her geometriyi kapsayan Bounding Box'ları ağaç şeklinde gruplar. Kutular birbirinin içine girebilir. Poligon ve LineString için zorunludur.
🔲 SP-GiST (Quad-Tree) — Satranç Tahtası
Mekanı kesişmeyen hücrelere böler. Noktalar kesişmediğinden ağaç çok daha sığ olur → Point (POI, adres) sorgularında GiST'ten hızlıdır.
📏 Pratik Kural
Poligon / LineString → GIST
Point (POI, adres) → SPGIST
⚠️ Az veride etkisi yok: Birkaç bin satırda PostgreSQL Seq Scan seçer. İndeksin gücü milyonlarca satırda ortaya çıkar.
Görsel Karşılaştırma
-- 1) GiST (R-Tree): Poligon ve Çizgiler için ZORUNLU CREATE INDEX IF NOT EXISTS idx_binalar_gist ON konya.osm_binalar USING GIST(geom); -- 2) SP-GiST (Quad-Tree): Noktalar (POI, adres) için DAHA HIZLI CREATE INDEX IF NOT EXISTS idx_poi_spgist ON konya.poi USING SPGIST(geom); -- Doğrulama: indeks gerçekten kullanılıyor mu? EXPLAIN ANALYZE SELECT id, ad FROM konya.poi WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326); -- ✓ Beklenen plan: "Bitmap Index Scan on idx_poi_spgist"
Kritik Soru: Neden her şeye GiST kurmuyoruz? ▼
GiST, kapsayıcı kutuları iç içe geçen klasörler gibi tutar — bu poligon/çizgi için kaçınılmaz. Ama milyonlarca nokta için ağaç gereksiz derinleşir. SP-GiST satranç tahtası mantığıyla noktayı tek bir hücreye düşürür → ağaç düzleşir → arama hızlanır.
Görev: İndeks Kapışması ▼
konya.poi üzerinde mevcut GiST indeksini düşürüp yerine SP-GiST kurun, aynı sorguyu EXPLAIN ANALYZE ile çalıştırın.
DROP INDEX IF EXISTS konya.idx_poi_geom; CREATE INDEX idx_poi_spgist ON konya.poi USING SPGIST(geom); EXPLAIN ANALYZE SELECT id, ad, kategori FROM konya.poi WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326);

4.1.2 — Bounding Box ve && Operatörü: İki Aşamalı Filtre
03 / 23
Kutu Matematiği
Her geometrinin etrafına gerilmiş sanal dikdörtgen — GiST indeksinin anladığı tek dil.
📦 Bounding Box (BBox) Nedir?
Geometriyi tam çevreleyen en küçük dikdörtgen. Kıvrımlı sınır veya dolambaçlı yol fark etmez — GiST içinde sadece bu kutu saklanır.
⚡ && — Kutu Kesişimi
İki geometrinin kutularının kesişip kesişmediğini sorar. Şekil hesabı yapmaz, saf indeks operasyonudur — ST_Intersects'ten yüzlerce kat hızlı.
🗺️ ST_MakeEnvelope
(MinX, MinY, MaxX, MaxY) köşelerini vererek hazır kutu üretir. Ekran görünümü, bölge filtresi, harita penceresi için doğrudan kullanılır.
⚠️ Kutu ≠ Şekil
&& kutu kesişimi onaylar, gerçek şekil kesişimini garanti etmez. Kesin sonuç için && + ST_Intersects birlikte kullanılır — önce kutu eleyici, sonra hassas filtre.
İki Aşamalı Filtre Akışı
-- ▶ Sadece kutu filtresi (saf indeks, çok hızlı) SELECT id, ad FROM konya.mahalleler WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326); -- ✓ Beklenen: ~100+ aday satır (kutu kesişimi) -- ▶ Kesin kesişim (&& sonra ST_Intersects ile hassas filtre) SELECT id, ad FROM konya.mahalleler WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326) AND ST_Intersects(geom, ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326)); -- ✓ Beklenen: kutudan az, gerçekten kesişen satırlar
Kritik Soru: && neden vazgeçilmezdir? ▼
GiST sadece Bounding Box'ları anlar; gerçek poligon matematiği yapan ST_Intersects tek başına indeks kullanamaz, tablo taraması yapar. && önce kutu ile veriyi %99 eler, indeks gücüyle çalışır. ST_Intersects sonra sadece kalan adaylar üzerinde çalışır → toplam maliyet milisaniyelere düşer.
Görev: Kutu / Şekil Farkını Ölç ▼
İki yöntemi EXPLAIN ANALYZE ile karşılaştırın; çıkan plan ve Execution Time'ı not edin:
-- 1) Saf kutu (ÇOK HIZLI) EXPLAIN ANALYZE SELECT id FROM konya.osm_binalar WHERE geom && ST_MakeEnvelope(32.45, 37.85, 32.55, 37.95, 4326); -- 2) Hassas kesişim (DAHA YAVAŞ) EXPLAIN ANALYZE SELECT id FROM konya.osm_binalar WHERE ST_Intersects(geom, ST_MakeEnvelope(32.45, 37.85, 32.55, 37.95, 4326));

4.1.3 — Partial ve Functional İndeksler: Hedefli Hız
04 / 23
Akıllı İndeksler
Tüm tabloyu indekslemek yerine — sadece kritik satırları ya da fonksiyon çıktısını indeksle.
🔪 Partial Index (Kısmi)
WHERE kategori='Eczane' koşuluyla sadece o satırları indeksler. 258 eczane için minik indeks; binlerce diğer POI'yi şişirmez. INSERT/UPDATE de sadece eczaneleri günceller.
⚙️ Functional Index (Fonksiyonel)
Fonksiyon sonucunu önceden hesaplayıp saklar. ST_Transform(geom, 3857) indekslenirse, sorgudaki her ST_Transform bu indeksi kullanır — dönüşüm tekrar yapılmaz.
📐 Ne Zaman Hangi?
Tablonun küçük alt kümesini sık sorguluyorsan → Partial.
Sorgularda aynı fonksiyonu tekrarlıyorsan → Functional.
💡 İkisi de normal indeksten küçük, dolayısıyla disk ve RAM'de hafif. Aşırı indeks yerine hedefli kalın.
Disk Maliyeti Karşılaştırması
❌ Tüm POI üzerinde GiST
14.000 satır × geom
~12 MB
✅ Partial (sadece Eczane)
258 satır × geom
~0.2 MB
-- 1) PARTIAL INDEX — Sadece eczaneleri indeksle CREATE INDEX idx_eczane_geom ON konya.poi USING SPGIST(geom) WHERE kategori = 'Eczane'; -- 2) FUNCTIONAL INDEX — ST_Transform önceden hesaplanır CREATE INDEX idx_bina_3857 ON konya.osm_binalar USING GIST(ST_Transform(geom, 3857)); -- 3) Doğrulama: indeks boyutunu görüntüle SELECT indexrelname, pg_size_pretty(pg_relation_size(indexrelid)) AS boyut FROM pg_stat_user_indexes WHERE schemaname = 'konya' AND (indexrelname LIKE 'idx_eczane%' OR indexrelname LIKE 'idx_bina_3857');
Kritik Soru: Partial İndeks ne kazandırır? ▼
Milyonlarca POI'nin %1'i "Eczane" ise tüm tabloyu indekslemek RAM, disk ve INSERT hızını sömürür. Partial indeks sadece o %1'i indeksler → arama uçar, alakasız INSERT'ler indeksi güncellemekle vakit kaybetmez.
Görev: Fonksiyonel İndeksi Yaz ▼
konya.osm_binalar tablosunda her binanın merkezini (ST_Centroid) indeksleyen fonksiyonel indeksi yazın.
CREATE INDEX idx_bina_merkez ON konya.osm_binalar USING GIST(ST_Centroid(geom));

4.1.4 — ANALYZE: Query Planner'ın Pusulası
05 / 23
Sorgu Pusulası
PostgreSQL sorgu planlarken veri dağılımını bilmek zorunda — ANALYZE bu istatistiği toplar.
🗺️ Query Planner Nedir?
Her SQL çalışmadan önce PostgreSQL en hızlı yolu seçer: "Index mi kullansam, Seq Scan mı?" Karar için tablonun veri profilini bilmek şarttır.
📊 ANALYZE Ne Yapar?
Tablodan örnek satırlar okur; sütun başına histogram (değer dağılımı), n_distinct ve null_frac istatistiklerini pg_statistic'e yazar.
⚠️ Ne Zaman Zorunlu?
Toplu yükleme (COPY, INSERT ... SELECT) sonrası. Eski boş tablo istatistikleriyle planner felç olur — yanlış plan seçer.
💡 autovacuum normalde ANALYZE'ı otomatik tetikler. Büyük toplu yüklemelerden hemen sonra manuel çalıştırmak güvenli alışkanlıktır.
▶ ADIM 1 — İstatistikleri topla
-- Toplu yüklemeden sonra ZORUNLU ANALYZE konya.osm_binalar; ANALYZE konya.mahalleler; ANALYZE konya.poi; -- ✓ Beklenen: Saniyeler içinde tamamlanır
▶ ADIM 2 — Toplanan istatistiği gör
SELECT attname AS sutun, n_distinct, null_frac, correlation FROM pg_stats WHERE schemaname='konya' AND tablename='osm_binalar'; -- ✓ Beklenen: tip, mahalle_id, kat_sayisi, geom satırları
▶ ADIM 3 — Plan değişti mi?
EXPLAIN ANALYZE SELECT * FROM konya.osm_binalar WHERE tip = 'residential'; -- ✓ Önce: Seq Scan (planner kötü tahmin) -- ✓ ANALYZE sonrası: Index Scan (doğru plan)
Kritik Soru: ANALYZE yapmazsam ne olur? ▼
1 milyon yeni satır eklediniz ve ANALYZE çalıştırmadınız → PostgreSQL hâlâ "tablo boş" sanır, küçük tablo planı seçer, Seq Scan zorlar. Sorgular dakikalarca sürer. ANALYZE = "Haritayı Güncelle!" komutudur.
Görev: Tüm Şemayı Pusulala ▼
VACUUM ANALYZE konya; tek komutu ile şemadaki tüm tabloların hem ölü satırlarını topla hem istatistiklerini güncelle. Ardından pg_stat_user_indexes'tan idx_scan değerlerini gözlemleyin.

4.1.5 — Ölü İndeks Avı: pg_stat_user_indexes
06 / 23
Ölü İndeks Avı
Kurulan her indeks yararlı mı? Yoksa disk yiyip INSERT hızını mı sömürüyor?
📊 pg_stat_user_indexes
PostgreSQL her indeksin kaç kez okunduğunu idx_scan sayacında tutar. Sunucu yeniden başlatılana dek birikir.
💀 idx_scan = 0 Ne Demek?
Bu indeks kurulduğundan beri hiç okunmadı. Disk kaplar, ama hiçbir sorguya katkı yapmadı.
🐢 Ölü İndeksin Gizli Maliyeti
Hiç okunmasa da her INSERT / UPDATE bu indeksi güncellemek zorundadır. Yazma hızını doğrudan yavaşlatır.
💡 Eğitim sonunda bu sorguyu çalıştırın. Listede çıkanları DROP INDEX ile temizleyin — veritabanı rahatlar.
▶ Ölü indeksleri bul
SELECT indexrelname AS indeks_adi, pg_size_pretty(pg_relation_size(indexrelid)) AS bosuna_kapladigi_yer, idx_scan AS kullanim_sayisi FROM pg_stat_user_indexes WHERE schemaname = 'konya' AND idx_scan = 0 ORDER BY pg_relation_size(indexrelid) DESC; -- ✓ Beklenen: Hiç kullanılmamış indekslerin listesi
▶ En çok kullanılan indeksler (sağlıklı)
SELECT indexrelname, idx_scan, pg_size_pretty(pg_relation_size(indexrelid)) AS boyut FROM pg_stat_user_indexes WHERE schemaname = 'konya' ORDER BY idx_scan DESC LIMIT 10; -- ✓ Beklenen: En çok okunmuş indeksler
▶ Ölü indeksi temizle
DROP INDEX IF EXISTS konya.idx_eski_kullanilmayan; -- Veya CONCURRENTLY ile (canlı sistemde): -- DROP INDEX CONCURRENTLY konya.idx_eski_kullanilmayan;
Kritik Soru: Hiç kullanılmayan indeksi neden silmeliyim? ▼
Sadece disk kaplamakla kalmaz! Tabloya her yeni veri (INSERT/UPDATE) geldiğinde PostgreSQL bu "işe yaramaz" indeksi de güncellemek zorundadır. Yazma performansı doğrudan etkilenir. Mümkün olan en az indeksle yola devam edin — disiplinli DBA, az ama doğru indeks kurar.
Görev: Çöpü Dışarı Çıkar ▼
Yukarıdaki SQL'i çalıştırın. idx_scan = 0 ile çıkan herhangi bir kendi kurduğunuz indeksi DROP INDEX ile silin (sistem indekslerine — primary key vb. — dokunmayın). Tekrar sorguya gözlemleyin: daha temiz bir liste.

4.2.0 — Güvenlik Katmanları ve Minimum Yetki Prensibi
07 / 23
5 Katman Güvenlik
PostgreSQL erişim kontrolü çok katmanlı bir piramittir. Her katman bir önceki üzerinde durur — biri çökerse diğeri devreye girer.
1️⃣ Network — pg_hba.conf
Hangi IP'den, hangi yöntemle (md5, scram, peer) bağlanılabilir? Veritabanına ulaşmadan önce kapıdaki güvenlik.
2️⃣ Rol / Kullanıcı — CREATE ROLE
Her kullanıcı bir rol. Roller gruplandırılır (fen_isleri, planlama). LOGIN, SUPERUSER, CREATEDB attribütleri.
3️⃣ Şema / Tablo — GRANT
Roller hangi şemayı (USAGE), hangi tabloyu (SELECT/INSERT/UPDATE/DELETE) görebilir? Nesne bazlı izin.
4️⃣ Satır — Row Level Security
Aynı tabloda, kullanıcıya göre farklı satır seti. ST_Intersects politikası ile mekansal filtre.
5️⃣ Sütun — Column Privileges
GRANT SELECT (ad, geom) ON ... ile hassas sütunları gizle. Maaş, TC kimlik gibi alanlar için.
Güvenlik Katmanları Piramidi
🎯 Minimum Yetki Prensibi
Her role sadece işi için gereken izin verilir. DELETE yetkisi olmayan kullanıcı kazara silemez; UPDATE yoksa veriyi bozamaz. Belediye ölçeğinde bu prensip ihlal edilirse — bir kullanıcı bile riskli kalır.
Kritik Soru: Neden 5 katman? Tek katman yeterli değil mi? ▼
Defense-in-depth prensibi: bir katman çökerse diğeri devreye girer. Örnek: Web uygulaması SQL injection ile hacklenirse (sütun seviyesi yetersiz), ama read-only kullanıcı (rol katmanı) kullanılıyorsa veri değişemez. RLS aktifse, hacklenen kullanıcı sadece kendi ilçe verisini görür. Her katman bir önceki güvensizleştiğinde son savunma.

4.2.1 — GRANT: Nesne İzinleri (USAGE, SELECT, INSERT...)
08 / 23
Nesne İzinleri
Şema, tablo, sütun — her nesne için ayrı bir izin katmanı.
📋 İzin Hiyerarşisi
Önce GRANT USAGE ON SCHEMA — şemaya girmek için şart. Sonra tabloya SELECT / INSERT / UPDATE / DELETE. USAGE olmadan tablo izni işe yaramaz.
🎯 Minimum Yetki
Her role sadece işi için gerekli izni verin. DELETE yoksa kazara silme imkansız. ALL PRIVILEGES sadece nesne sahibine yakışır.
🔮 Gelecek Tablolar
GRANT ON ALL TABLES sadece mevcut tabloları kapsar. Sonradan eklenen tablolar için ALTER DEFAULT PRIVILEGES şarttır.
🔍 İzinleri Sorgula
\dp konya.osm_binalar (psql) veya information_schema.role_table_grants sorgusu.
▶ ADIM 1 — Salt-Okuyucu Rol Kur
-- Rol oluştur (LOGIN yok = grup rolü) CREATE ROLE okuyucu_rol; -- ZORUNLU: Şemaya giriş izni GRANT USAGE ON SCHEMA konya TO okuyucu_rol; -- Tablo bazlı SELECT GRANT SELECT ON konya.mahalleler TO okuyucu_rol; GRANT SELECT ON konya.poi TO okuyucu_rol; -- ✓ Beklenen: GRANT
▶ ADIM 2 — Yazma Yetkisi (DELETE Hariç)
-- Fen İşleri: yolları okusun, ekleyebilsin, güncelleyebilsin CREATE ROLE fen_isleri_rol; GRANT USAGE ON SCHEMA konya TO fen_isleri_rol; GRANT SELECT, INSERT, UPDATE ON konya.osm_yollar TO fen_isleri_rol; -- Kaza kayıtlarını ekleyebilsin ama silemesin GRANT USAGE ON SCHEMA analiz TO fen_isleri_rol; GRANT SELECT, INSERT ON analiz.kazalar TO fen_isleri_rol;
▶ ADIM 3 — Gelecek Tablolar İçin Default
-- Bundan sonra konya şemasına eklenen TÜM tablolar -- otomatik olarak okuyucu_rol'e SELECT verir ALTER DEFAULT PRIVILEGES IN SCHEMA konya GRANT SELECT ON TABLES TO okuyucu_rol; -- ✓ Beklenen: ALTER DEFAULT PRIVILEGES
Kritik Soru: USAGE olmadan SELECT yeterli mi? ▼
Hayır. GRANT SELECT ON konya.mahalleler TO okuyucu_rol versek bile GRANT USAGE ON SCHEMA konya olmadan kullanıcı şemaya giremez — sorgu "şema yok veya erişim yok" hatası verir. Her zaman önce USAGE, sonra tablo izni.
Görev: İzleme Rolü Kur ▼
izleme_rol adında bir rol oluşturun. konya.osm_binalar ve konya.mahalleler tablolarında yalnızca SELECT yetkisi verin. ALTER DEFAULT PRIVILEGES ile sonradan eklenecek konya tablolarını da kapsayın. Doğrulama: \dp konya.*

4.2.2 — Rol Üyeliği ve İzin Devri (WITH GRANT OPTION)
09 / 23
Rol Üyeliği
Kullanıcıyı gruba ekle — gruptaki tüm izinler anında devralınır. Tek kişiye 50 GRANT yerine 1 GROUP membership.
👥 Rol Hiyerarşisi
Bir role başka bir rolü üye yapabilirsin. GRANT fen_isleri_rol TO ali → Ali, rolün tüm izinlerini devralır. Onlarca kullanıcıyı tek rol üzerinden yönet.
🔑 WITH GRANT OPTION
İzni alan kişi başkasına da devredebilir. Tehlikeli — yetki zinciri kontrolden çıkabilir. Yalnızca güvenilen birim yöneticilerine verin.
⛔ REVOKE CASCADE
WITH GRANT OPTION ile dağıtılan izinleri geri almak için CASCADE şart. Yoksa sadece doğrudan verilen izin gider; devredilenler kalır.
🧪 SET ROLE ile Test
SET ROLE fen_isleri_rol; ile o role bürün, izinleri test et. RESET ROLE; ile geri dön. Production öncesi zorunlu adım.
▶ ADIM 1 — Grup Rol ve Kullanıcılar
-- Grup rolü (Slayt 8'de zaten oluşturduysan bu satırı atla) -- CREATE ROLE fen_isleri_rol; GRANT USAGE ON SCHEMA konya TO fen_isleri_rol; GRANT SELECT, INSERT, UPDATE ON konya.osm_yollar TO fen_isleri_rol; -- Bireysel kullanıcılar (LOGIN var) CREATE ROLE ali LOGIN PASSWORD 'ali2024'; CREATE ROLE ayse LOGIN PASSWORD 'ayse2024'; -- Tek seferde gruba ekle GRANT fen_isleri_rol TO ali, ayse; -- ✓ Ali ve Ayşe artık fen_isleri yetkilerinin tümüne sahip
▶ ADIM 2 — WITH GRANT OPTION (Devir)
-- Önce planlama_rol oluştur (yoksa SET ROLE başarısız olur) CREATE ROLE planlama_rol; CREATE ROLE mehmet LOGIN PASSWORD 'mehmet2024'; -- ⚠️ USAGE olmadan başka rol adına GRANT yapılamaz GRANT USAGE ON SCHEMA konya TO planlama_rol; -- Planlama müdürü başkalarına da izin verebilsin GRANT SELECT ON konya.ilce_sinirlari TO planlama_rol WITH GRANT OPTION; -- Planlama rolüne sahip kullanıcı bu izni başkasına verebilir SET ROLE planlama_rol; GRANT SELECT ON konya.ilce_sinirlari TO mehmet; RESET ROLE;
▶ ADIM 3 — REVOKE ve CASCADE
-- Sadece ali'yi rolden çıkar (ayşe etkilenmez) REVOKE fen_isleri_rol FROM ali; -- WITH GRANT OPTION zincirini de temizle REVOKE SELECT ON konya.ilce_sinirlari FROM planlama_rol CASCADE; -- ✓ Mehmet'in dolaylı izni de iptal olur
Kritik Soru: Kullanıcıya mı, role mi izin verilmeli? ▼
Her zaman role verin, kullanıcıya değil. Ali ayrılırsa REVOKE fen_isleri_rol FROM ali yeterli — onlarca tabloyu tek tek temizlemek yok. Yeni gelen Veli'ye sadece GRANT fen_isleri_rol TO veli. Rol = yetki şablonu.
Görev: Birim Yönetimi Senaryosu ▼
imar_rol oluşturun: konya.mahalleler SELECT, konya.ilce_sinirlari SELECT + INSERT yetkisi. İki test kullanıcısı oluşturup role ekleyin. Birini REVOKE ile çıkarıp SET ROLE imar_rol ile diğerinin izinlerini doğrulayın.

4.2.3 — Mekansal Row Level Security (RLS)
10 / 23
Bölge Bazlı Kısıt
Meram personeli yalnızca Meram'ı görür — otomatik, şeffaf, kullanıcı farkında bile değil.
🔒 RLS Nedir?
Row Level Security — her kullanıcı sorgusuna otomatik WHERE koşulu eklenir. SELECT * FROM tablo dese bile arka planda WHERE ... uygulanır.
🗺️ Mekansal RLS
Politikanın USING koşuluna ST_Intersects(geom, ilce_sinirlari) yazılır. Kullanıcı sadece kendi ilçesindeki kayıtları görür.
⚡ Performans Notu
Her sorguda ST_Intersects çalışır → GiST indeksi zorunlu. İndeks varsa etki milisaniyeler düzeyindedir.
🧪 SET ROLE ile Doğrula
SET ROLE meram_calisani; ile mevcut oturumda o kullanıcı gibi davran, satır sayısını kontrol et, sonra RESET ROLE.
RLS Görselleştirmesi
-- ▶ ADIM 1: Rol + temel yetki CREATE ROLE meram_calisani LOGIN PASSWORD 'meram2024'; GRANT USAGE ON SCHEMA konya TO meram_calisani; GRANT SELECT ON konya.osm_binalar TO meram_calisani; GRANT SELECT ON konya.ilce_sinirlari TO meram_calisani; -- ▶ ADIM 2: RLS aç + politika ALTER TABLE konya.osm_binalar ENABLE ROW LEVEL SECURITY; CREATE POLICY meram_only ON konya.osm_binalar AS PERMISSIVE FOR SELECT TO meram_calisani USING (ST_Intersects(geom, (SELECT geom FROM konya.ilce_sinirlari WHERE ad = 'Meram'))); -- ▶ ADIM 3: Test SELECT count(*) FROM konya.osm_binalar; -- 18279 (admin) SET ROLE meram_calisani; SELECT count(*) FROM konya.osm_binalar; -- ~25 (RLS aktif) RESET ROLE;
Kritik Soru: RLS performansı etkiler mi? ▼
Evet, her sorguya otomatik ST_Intersects filtresi eklenir. Ama geom kolonunda GiST indeksi varsa etki milisaniyeler düzeyindedir. Önce indeks kontrol: \d konya.osm_binalar komutunda GiST indeksini görmelisiniz.
Görev: Selçuklu Politikası ▼

Tablolar + indeksler + sequence + fonksiyon + PostGIS uzantı kaydı + view. Sadece şema için -s, sadece veri için -a.
⏰ Otomatik Yedek (cron)
Linux'ta crontab ile her gece yedek + retention. Belediye sistemleri için kritik — manuel yedek unutulur.
📘 pg_dump vs pg_dumpall
pg_dump tek bir veritabanını yedekler. pg_dumpall tüm cluster'ı (tüm DB + roller + grant + tablespace) yedekler. Standart pratik: günlük pg_dump + haftalık pg_dumpall.
⚠️ NEREDE çalışır? pg_dump ve pg_restore terminal / CMD komutlarıdır — DBeaver SQL editöründe ÇALIŞMAZ. Docker için docker exec ile gönderilir.
▶ Yedekleme (Terminal / CMD)

▶ Geri Yükleme

4.3.3 — PITR: Felaket Anında Zamanı Geri Al
14 / 23
Zaman Makinesi
"Dün saat 14:32'deki duruma geri dön" — saniyeler içinde.
🔑 Analoji: WAL = banka dekontu defteri. Her hareket bir dekont; defter yanmadıysa bakiyeni dakikası dakikasına yeniden çıkartabilirsin.
⏪ PITR Nedir?
Point-In-Time Recovery. Yanlış DELETE veya hatalı migrasyon sonrası DB'yi tam o ana döndürür. Granülerlik saniye seviyesinde.
📜 WAL (Write-Ahead Log)
Her INSERT/UPDATE/DELETE diske işlenmeden önce WAL'a yazılır. Base backup + WAL arşivi birlikte → herhangi bir ana gidiş.
⚠️ Ön Koşul
wal_level = replica, archive_mode = on önceden ayarlı olmalı. Felaket sonrası açmak çalışmaz — WAL kaybolur.
🔁 Geri Dönüş Akışı
1. Base backup'ı geri yükle
2. recovery.signal oluştur
3. restore_command ile WAL uygula
4. recovery_target_time'da dur, promote
WAL Zaman Çizelgesi (PITR)
-- ▶ 1) ÖNCEDEN: postgresql.conf — WAL arşivlemeyi AÇ wal_level = replica archive_mode = on archive_command = 'cp %p /arsiv/wal/%f' -- ⚠️ Değişiklik sonrası: pg_ctl restart (RELOAD yetmez) -- ▶ 2) HAFTALIK: Base backup pg_basebackup -D /yedek/base_$(date +%F) -Ft -z -P -U replicator -- ▶ 3) FELAKET ANI: 14:31:59'a geri dön -- a) PostgreSQL'i durdur, data dizinini sil -- b) Base backup'ı aç tar -xzf /yedek/base_2026-05-20/base.tar.gz \ -C /var/lib/postgresql/data -- c) recovery.signal (boş dosya yeter) touch /var/lib/postgresql/data/recovery.signal -- d) postgresql.conf — recovery parametreleri restore_command = 'cp /arsiv/wal/%f %p' recovery_target_time = '2026-05-20 14:31:59' recovery_target_action = 'promote' -- e) Başlat → otomatik recovery, hedefte durur pg_ctl start
Kritik Soru: WAL nereye yazılır, ne kadar tutulur? ▼
WAL 16 MB segmentler halinde $PGDATA/pg_wal/'a yazılır. Segment dolduğunda archive_command tetiklenir → dosya arşive (yerel disk, NFS, S3) kopyalanır. Granülerlik: her commit bir WAL kaydı → saniyenin altına gidilebilir.

📜 MVCC Mantığı
UPDATE/DELETE satırı diskten silmez. "Ölü (dead tuple)" işaretler. Yeni versiyon yanına yazılır. Eşzamanlı okumalar kilitlenmeden çalışır.
🧟 Ölü Satır = Hayalet
VACUUM gelene kadar diskte durur. Normal SELECT görmez, ama oradadır — doğru araçla okunur.
🔌 pg_dirtyread Eklentisi

🛡️ Kurtarma Penceresi = autovacuum

BEGIN; DELETE FROM konya.poi WHERE kategori = 'Eczane'; -- 258 satır gitti. COMMIT edilmedi — henüz geri alınabilir! -- ▶ 3) KURTARMA: VACUUM çalışmadan önce ölü satırları oku -- ⚠️ Kolon tipleri tablodakiyle TAM eşleşmeli (varchar(200) vs text farkı hata verir) SELECT * FROM pg_dirtyread('konya.poi'::regclass) AS t(id int, ad varchar(200), kategori varchar(100), geom geometry) WHERE kategori = 'Eczane'; -- ✓ Beklenen: 258 satır (henüz disk temizlenmedi) -- ▶ 4) Geri yaz: çıktıyı tabloya tekrar INSERT et INSERT INTO konya.poi (id, ad, kategori, geom) SELECT id, ad, kategori, geom FROM pg_dirtyread('konya.poi'::regclass) AS t(id int, ad varchar(200), kategori varchar(100), geom geometry) WHERE kategori = 'Eczane' ON CONFLICT (id) DO NOTHING; ROLLBACK; -- Eğitim bitince ROLLBACK ile geri al -- ▶ 5) KURTARMA PENCERESİNİ GENİŞLET: autovacuum'u geçici durdur -- Ölü satırlar yalnızca autovacuum (veya manuel VACUUM) tabloya -- dokununcaya kadar pg_dirtyread ile okunabilir kalır. MVCC: Multi-Version Concurrency Control — satırın eski + yeni versiyonları yan yana, kilit yok.
Dead tuple: "Silindi" işaretli ama diskte duran eski satır.
VACUUM: Dead tuple toplayıp yer açan bakım (otomatik = autovacuum).

4.3.6 — Logical Decoding: WAL'i Okuyup Ters Çevirmek
17 / 23
WAL'i Tersine Oku
"Dün tam olarak hangi UPDATE'ler çalıştı?" sorusunun cevabı.
🔑 Analoji: WAL = ham veri (PostgreSQL iç dili). Logical Decoding = bu ham veriyi "Türkçe" okunabilir kayıtlara çeviren tercüman.
🔄 Logical Decoding Nedir?
WAL'in ham byte akışını okunabilir kayıt akışına çevirir. Her transaction'ın eski/yeni değerleri tek tek dışarı alınır.
🧩 Replication Slot
WAL'in geçmişe dönük korunması için tutamaç. Slot ilerlemediği sürece PostgreSQL o noktadan sonraki WAL'leri silmez. Slot YOK = geçmiş WAL YOK.
🔌 Decoder Plugin
WAL byte'larını okunur formata çeviren modül. wal2json en yaygın (JSON çıktı). pgoutput standart replication için.
📘 Temel Kavram
Undo SQL: Bir UPDATE'in tersi. INSERT'in tersi DELETE, DELETE'in tersi INSERT.
LSN (Log Sequence Number): WAL içinde bayt offset. Slot bunu tutar.
WAL'i okumak = geçmişi okumak, veriyi değil.
Kavramsal Akış
UPDATE → WAL (ham byte)
↓ Replication Slot akışı tutar
Decoder Plugin → JSON kayıt
↓ Script OLD ve NEW'i ayıklar
Tersleyici → UNDO SQL
⚠️ Kritik: Replication slot olay öncesi kurulmalı. Slot yoksa eski WAL'ler silinmiştir — geçmişe dönülemez. Bu yüzden audit trigger (sonraki slayt) bir B planı olarak kurulur.
Slot yoksa eski WAL'ler neden silinir? ▼
PostgreSQL WAL'leri sınırsız tutamaz — disk dolar. "Hiç kimse okumak istemiyor" olan WAL silinir. Slot, "ben bunu okumak istiyorum" işareti koyar; primary o WAL'i silmeye cesaret edemez. Slot yoksa retention wal_keep_size ile sınırlıdır — varsayılanı 0, yani ekstra WAL hiç saklanmaz; eski segmentler checkpoint sonrası geri dönüştürülür. Logical Decoding retrospektif değil proaktif: kurmadıysan geçmişe dönemez.

4.3.8 — Audit Trigger: Her Değişikliğin Kara Kutusu
19 / 23
Geleceğe Yatırım
🔑 Audit log = güvenlik kamerası. Hiçbir değişiklik kameraya yansımadan veritabanına işlenmiyor.
🎯 Neden Trigger?
Eklenti veya WAL arşivi gerektirmez. Herhangi bir PostgreSQL kurulumunda çalışır. Tüm değişiklikler ayrı bir audit_log tablosuna düşer — silinen veri asla kaybolmaz.
📊 OLD ve NEW Nedir?
Trigger fonksiyonu içinde iki özel değişken: OLD = değişiklikten önceki satır, NEW = sonraki satır. DELETE'de sadece OLD, INSERT'te sadece NEW. row_to_json() ile JSONB → tek audit tablosu tüm tablolar için.
🔁 BEFORE vs AFTER
BEFORE: İşlemden önce; değişikliği iptal/değiştirebilir (doğrulama için).
AFTER: İşlem bitince çalışır; sadece kayıt → audit log için doğru seçim.
📈 Hacim Kontrolü
Audit tablosu sürekli büyür. Çözüm: partitioning (aylık) veya rolling retention (90 günden eskileri sil).
-- ▶ 1) Audit tablosu — tüm şema için tek tablo CREATE TABLE konya.audit_log ( id BIGSERIAL PRIMARY KEY, tablo TEXT NOT NULL, islem TEXT NOT NULL, -- INSERT/UPDATE/DELETE eski_veri JSONB, -- OLD satır yeni_veri JSONB, -- NEW satır kullanici TEXT DEFAULT current_user, zaman TIMESTAMPTZ DEFAULT NOW() ); CREATE INDEX idx_audit_tablo_zaman ON konya.audit_log (tablo, zaman DESC); -- ▶ 2) Generic trigger function (tüm tablolar için tek) CREATE OR REPLACE FUNCTION konya.audit_trigger() RETURNS TRIGGER AS $$ BEGIN INSERT INTO konya.audit_log (tablo, islem, eski_veri, yeni_veri) VALUES ( TG_TABLE_NAME, TG_OP, CASE WHEN TG_OP IN ('UPDATE','DELETE') THEN row_to_json(OLD)::jsonb END, CASE WHEN TG_OP IN ('UPDATE','INSERT') THEN row_to_json(NEW)::jsonb END ); RETURN COALESCE(NEW, OLD); END; $$ LANGUAGE plpgsql; -- ▶ 3) Trigger'ı tabloya bağla CREATE TRIGGER trg_audit_poi AFTER INSERT OR UPDATE OR DELETE ON konya.poi FOR EACH ROW EXECUTE FUNCTION konya.audit_trigger(); -- ▶ 4) KURTARMA: Silinen kaydı audit'ten geri getir INSERT INTO konya.poi SELECT (jsonb_populate_record(NULL::konya.poi, eski_veri)).* FROM konya.audit_log WHERE tablo = 'poi' AND islem = 'DELETE' AND zaman > NOW() - INTERVAL '1 hour';
Audit'ten tek bir satırı nasıl geri getiririm? ▼
jsonb_populate_record JSONB'yi tablo satırına çevirir. id=42 silindiyse: INSERT INTO konya.poi SELECT (jsonb_populate_record(NULL::konya.poi, eski_veri)).* FROM konya.audit_log WHERE tablo='poi' AND islem='DELETE' AND (eski_veri->>'id')::int = 42; JSONB'deki tüm sütunlar otomatik eşleşir.
Görev: poi tablosuna audit kur ▼
1) Üstteki 3 SQL bloğunu sırayla çalıştır.
2) UPDATE konya.poi SET ad = 'TEST' WHERE id = 1;
3) SELECT * FROM konya.audit_log ORDER BY zaman DESC LIMIT 5; — JSON'u gör.
4) Adım 4'teki kurtarma sorgusuyla eski haline döndür.

4.4.1 — PgBouncer: Bağlantı Maliyeti ve Havuzlama
20 / 23
Neden PgBouncer?
🔑 Restoran rezervasyonu. 1000 müşteri ama 20 masa. PgBouncer = hostes; gelene masa ayarlar, biri kalkınca diğerini oturtur.
🎯 Problem: Her Bağlantı Bir Process
PostgreSQL'de her bağlantı OS'de ayrı bir process açar. Process başına ~10 MB RAM. 1000 eş zamanlı kullanıcı = 10 GB sadece bağlantıya gider → sunucu çöker.
💀 Idle Connection Tuzağı
Uygulama açık ama o an iş yapmayan bağlantı = idle connection. Process RAM'de yaşamaya devam eder. Web uygulamaları bazen yüzlerce idle bağlantı tutar — tüm bu RAM boşa gider.
🏊 Havuzlama Mantığı
PgBouncer, uygulama ile PostgreSQL arasında durur. 1000 web isteği PgBouncer'a gelir, PgBouncer sadece 20 gerçek bağlantı açar ve paylaştırır. PostgreSQL 20 process görür → rahat çalışır.
🛤️ PgBouncer'ın Konumu
Uygulama artık PostgreSQL'e (5432) değil PgBouncer'a (6432) bağlanır. PgBouncer arka planda gerçek bağlantıyı yönetir — uygulama kodu değişmez.
📘 Temel Kavram
Connection: Uygulamanın PostgreSQL'e açtığı kanal. Her biri sunucuda bir UNIX süreci başlatır.
Process: OS seviyesinde RAM tüketen iş birimi (~10 MB / bağlantı).
Idle Connection: Açık ama iş yapmayan bağlantı — process RAM'i kullanmaya devam eder.
Connection Pooler: Az sayıda gerçek bağlantıyla çok sayıda istemciyi karşılayan ara katman.
Bağlantı Akış Diyagramı
PgBouncer Olmadan Ne Olur? ▼
Doğrudan 500+ bağlantı açılırsa PostgreSQL yavaşlar, sonra FATAL: sorry, too many clients already hatası verir. Uygulamalar bağlanamaz, tüm sistem durur. CBS web haritaları yüklenemez. PgBouncer bu senaryoyu tamamen ortadan kaldırır.
Görev: max_connections'ı kontrol et ▼
Mevcut PostgreSQL ayarını gör: SHOW max_connections; (genellikle 100). SELECT count(*) FROM pg_stat_activity; ile aktif bağlantı sayısını gör. Eğer 80'i geçtiğiniz olduysa, PgBouncer artık zorunlu.

4.4.2 — PgBouncer Kurulum ve pool_mode Seçimi
21 / 23
Hangi Mod?
🔑 3 farklı servis modeli: 'Müşteri masadan kalkmadan yemek alır' (session), 'Yemek arası masa boşalır' (transaction), 'Her lokmada masa değişir' (statement).
🟢 session mode — En Güvenli
Bağlantı kapanana kadar aynı PG bağlantısı ayrılır. SET, PREPARE, advisory lock, geçici tablo — hepsi sorunsuz. Dezavantaj: havuz uzun süre meşgul.
🟡 transaction mode — CBS İçin İdeal
Her COMMIT/ROLLBACK sonrası bağlantı havuza döner. Bağlantı sadece işlem süresince meşgul → verimlilik yüksek. CBS web haritaları ve REST API'ler için ideal.
🔴 statement mode — Çok Agresif
Her SQL biter bitmez bağlantı havuza döner. En yüksek verimlilik ama kırılgan: multi-statement transaction, PREPARE/EXECUTE, SET — hiçbiri çalışmaz. Neredeyse hiç kullanılmaz.
⚠️ Sık Hata: Prepared Statement
Transaction mode'da PREPARE ... EXECUTE kullanırsanız hata alırsınız (farklı bağlantıya düşer). Çözüm: ya session mode ya da uygulama tarafında prepared statement'ı devre dışı bırak.
▶ ADIM 1 — pgbouncer.ini
[databases] konya_egitim = host=localhost port=5432 dbname=konya_egitim [pgbouncer] listen_port = 6432 listen_addr = * auth_type = scram-sha-256 auth_file = /etc/pgbouncer/userlist.txt pool_mode = transaction max_client_conn = 1000 ; toplam istemci default_pool_size = 20 ; DB başına gerçek PG bağlantısı min_pool_size = 5 server_idle_timeout = 600 admin_users = egitim ; SHOW POOLS/CLIENTS yetkisi
▶ ADIM 2 — userlist.txt + başlat
# /etc/pgbouncer/userlist.txt "egitim" "SCRAM-SHA-256$4096:..." # Başlat sudo systemctl restart pgbouncer sudo systemctl status pgbouncer # ✓ Beklenen: active (running)
▶ ADIM 3 — Bağlan ve izle
# Port 6432 üzerinden bağlan psql -h localhost -p 6432 -U egitim konya_egitim # Havuz durumunu izle (pgbouncer admin DB) psql -h localhost -p 6432 -U egitim pgbouncer SHOW POOLS; -- her DB için havuz doluluğu SHOW CLIENTS; -- bağlı istemciler SHOW SERVERS; -- gerçek PG bağlantıları SHOW STATS; -- toplam istek / saniye
max_client_conn Değerini Nasıl Seçerim? ▼
Kural: default_pool_size × 10 MB ≤ PostgreSQL'e ayrılan RAM. Örnek: 4 GB PG RAM → max 400 MB bağlantı → default_pool_size = 40. max_client_conn ise bu havuzu paylaşan uygulama sayısı; 500–2000 arası güvenli. İstemci sayısı çok, gerçek bağlantı az mantığı.
Görev: PgBouncer Kur ve Bağlan ▼
1) sudo apt install pgbouncer (Ubuntu) veya Docker imajı.
2) /etc/pgbouncer/pgbouncer.ini'yi yukarıdaki örnekle düzenle.
3) userlist.txt'e SCRAM hash ekle.
4) systemctl restart pgbouncer.
5) psql -p 6432 -U egitim konya_egitim.
6) SHOW POOLS; ile havuz doluluk oranını izle — production'da bu sayı 0'dan yüksek olmalı.
