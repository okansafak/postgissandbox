2.1 — Veri Yükleme (CSV ve COPY)
02 / 22
CSV'yi Tek Komutla Yükle
Excel/CSV dosyasındaki binlerce satırı tek tek INSERT yazmadan, saniyeler içinde tabloya aktar.
🚚 COPY = Nakliye Kamyonu
INSERT kutuları tek tek taşır. COPY hepsini tek seferde boşaltır — yüzlerce kat hızlı.
📁 \copy vs COPY
\copy → dosya senin bilgisayarında. COPY → dosya sunucuda. Çoğu zaman \copy kullanırsın.
🔤 ENCODING 'UTF8'
Türkçe karakterler (ğ, ş, ı, ç) bozulmasın diye şart. Unutursan "Ã§" gibi çöp görürsün.
💡 Sıra önemli: Önce tabloyu oluştur (sütunlar CSV ile aynı sırada), sonra veriyi yükle.
-- 1️⃣ Önce hedef tabloyu oluştur -- Sütunlar CSV başlığıyla aynı sırada olmalı CREATE TABLE IF NOT EXISTS konya.nufus_copy ( yil INT, il VARCHAR(50), ilce VARCHAR(100), ilce_id INT, nufus INT ); -- 2️⃣ CSV'yi yükle (\copy = senin bilgisayarındaki dosya) -- FORMAT csv → virgülle ayrılmış dosya -- HEADER → ilk satır başlık, atla -- ENCODING → Türkçe karakterler için UTF8 \copy konya.nufus_copy FROM 'D:\nufus_copy.csv' WITH (FORMAT csv, HEADER, ENCODING 'UTF8') -- 3️⃣ Geldi mi? Kontrol et SELECT * FROM konya.nufus_copy LIMIT 10;
COPY Neden INSERT'ten Çok Hızlı? ▼
INSERT her satır için SQL'i ayrıştırır, kurallara bakar, log tutar — satır başına çok iş. COPY ise bu adımları toplu yapıp veriyi doğrudan diske yazar. 100.000 satır: INSERT ile dakikalar, COPY ile saniyeler.
Görev: Türkçe Karakter Testi ▼
1) CSV'yi yükle ve kontrol et:
SELECT * FROM konya.nufus_copy LIMIT 10;

2) İlçe adlarına bak — "Selçuklu", "Çumra" düzgün mü? Eğer "SelÃ§uklu" gibi görünüyorsa ENCODING 'UTF8' parametresini unutmuşsundur.

2.1 — ogr2ogr ile Veri Taşıma
03 / 22
Harita Dosyası Çevirici
ogr2ogr, harita formatları arasında çeviri yapan evrensel tercüman. Shapefile, GeoJSON, KML, DXF → hepsini PostGIS'e taşır.
🔄 Format + Koordinat Çevirisi
Tek komutta hem formatı çevirir hem -t_srs ile koordinat sistemini dönüştürür.
⚠️ Nerede Çalıştırılır?
QGIS kurunca GDAL ile gelir. Komutları OSGeo4W Shell'de çalıştır — psql içinde ÇALIŞMAZ.
🆚 ogr2ogr mu COPY mu?
Harita dosyası (SHP/GeoJSON) → ogr2ogr. Düz CSV → COPY (çok daha hızlı).
💡 Windows ipucu: Komut çok satırlı; satır sonundaki ^ işareti "komut devam ediyor" demek.
# Shapefile → PostGIS (koordinat dönüşümü ile) # -f : Hedef Format (PostgreSQL vb.) # -nln : Yeni Katman/Tablo Adı (New Layer Name) # -lco : Katman Ayarları (Burada şema belirtir: SCHEMA=konya) # -nlt : Geometri Tipi (PROMOTE_TO_MULTI Poligonları MultiPolygon yapar) # -t_srs : Hedef Koordinat Sistemi (Target SRS) # -overwrite : Tablo zaten varsa üzerine yaz # -progress : Yükleme ilerlemesini (%) gösterir ogr2ogr -f "PostgreSQL" ^ PG:"host=localhost port=5442 dbname=konya_egitim user=egitim password=egitim2024" ^ D:\data\mahalleler.shp ^ -nln mahalleler_tkgm ^ -lco SCHEMA=konya ^ -nlt PROMOTE_TO_MULTI ^ -t_srs EPSG:4326 ^ -overwrite ^ -progress # Sadece belirli alanları aktar (-sql ile veriyi yoldayken filtrele) ogr2ogr -f "PostgreSQL" ^ PG:"host=localhost port=5442 dbname=konya_egitim user=egitim password=egitim2024" ^ D:\data\mahalleler.shp ^ -nln mahalleler_secili ^ -lco SCHEMA=konya ^ -nlt PROMOTE_TO_MULTI ^ -sql "SELECT id, text AS mahalle_adi, source_d_1 AS ilce_adi FROM mahalleler" ^ -t_srs EPSG:4326 ^ -overwrite ^ -progress
Hangi Parametre Ne İşe Yarar? ▼
-nln → tablo adı  |  -lco SCHEMA=konya → hangi şemaya
-t_srs EPSG:4326 → koordinatı 4326'ya çevir
-nlt PROMOTE_TO_MULTI → Polygon'u MultiPolygon yap (karışık geometrilerde hata önler)
-overwrite → tablo varsa sil baştan yaz
Görev: ogr2ogr ile GeoJSON Yükle ▼
1) OSGeo4W Shell'i aç (psql değil!).
2) Yukarıdaki ilk komutu kopyala, .shp yerine mahalleler.geojson, -nln değerini mahalleler_test yap.
3) Çalıştır, sonra psql'de doğrula: SELECT COUNT(*) FROM konya.mahalleler_test;

2.1 — Veri Temizliği (Regex ve UPDATE)
04 / 22
Metni Standartlaştır
"Fakılar Mah." mı "Fakılar Mahallesi" mi "Fakılar" mı? Aynı yer, üç farklı yazım — JOIN bunları eşleştiremez.
❌ Sorun: JOIN Patlıyor
İki tablo "Fakılar Mah." ≠ "Fakılar" yüzünden eşleşmez. Tek harf farkı bile JOIN'i boş döndürür.
🔍 Çözüm: regexp_replace
Akıllı bul-değiştir. Bir kalıba uyan kısmı siler/değiştirir. " Mah." ekini hepsinden kırpar.
⚠️ Önce SELECT, sonra UPDATE
UPDATE kalıcıdır, geri alınamaz. Her zaman SELECT ile sonucu gör, doğruysa UPDATE yap.
💡 Kalıp: ' Mah\.$' = boşluk + "Mah" + nokta, satır sonunda ($ = son).
-- 1️⃣ ÖNCE SELECT: Temizlenmiş hali nasıl görünüyor? -- ' Mah\.$' = sondaki " Mah." ekini bul, sil SELECT ad AS orjinal, regexp_replace(ad, ' Mah\.$', '') AS temiz FROM konya.mahalleler LIMIT 5; -- 2️⃣ Baş/son boşlukları da temizle (TRIM) SELECT TRIM(regexp_replace(ad, ' Mah\.$', '')) AS temiz FROM konya.mahalleler LIMIT 5; -- 3️⃣ Asıl tabloyu bozmamak için bir KOPYA üzerinde çalış CREATE TABLE konya.mahalleler_temiz AS SELECT id, ad FROM konya.mahalleler; -- 4️⃣ Sonuç doğruysa kopyada UPDATE ile kalıcı yap UPDATE konya.mahalleler_temiz SET ad = TRIM(regexp_replace(ad, ' Mah\.$', '')); -- 5️⃣ Kontrol: artık " Mah." kalmadı mı? (boş dönmeli) SELECT ad FROM konya.mahalleler_temiz WHERE ad LIKE '% Mah.';
Kalıptaki Sembolleri Çöz ▼
$ → satırın sonu (sadece sondaki "Mah." gitsin, ortadaki kalsın)
\. → gerçek nokta karakteri (kaçış olmadan . "herhangi bir karakter" demek)
'' → bulunanı boşlukla değil, hiçbir şeyle değiştir = sil
Görev: Güvenli Temizlik Akışı ▼
1) 1. sorguyu çalıştır — orjinal ve temiz sütunlarını yan yana karşılaştır.

2) Doğruysa 3. (kopya oluştur) ve 4. (UPDATE) sorguları sırayla çalıştır.

3) 5. sorgu boş dönmeli — boş dönüyorsa temizlik başarılı. Asıl konya.mahalleler tablosu hiç bozulmadı.

2.2 — ST_Simplify ve ST_Smooth
05 / 22
Sadeleştir ya da Yumuşat
Çok detaylı sınırlar haritayı yavaşlatır. İki çözüm: gereksiz noktaları at, ya da köşeleri yuvarla.
📉 ST_Simplify (Nokta Azalt)
Şekli koruyarak gereksiz noktaları siler. Yunak ilçesi ~%75 hafifler. Harita daha hızlı çizilir.
🌊 ST_ChaikinSmoothing (Yumuşat)
Sivri köşeleri yuvarlar — el çizimi gibi. Tam tersi etki: geometri büyür, detay artar.
⚠️ Tolerans = DERECE (4326'da)
0.01 ≈ 1.1 km. 10.0 verirsen koca şehri yutar! Metre istiyorsan önce 3857'ye çevir.
💡 Güvenli sürüm: ST_SimplifyPreserveTopology şekli yırtmaz, poligonu çökertmez. Düz ST_Simplify bazen NULL döndürebilir.
-- 📉 1. SADELEŞTİR: Geometri ne kadar hafifledi? -- length(ST_AsText(...)) = geometrinin "ağırlığı" (karakter) SELECT ad, length(ST_AsText(geom)) AS orjinal_agirlik, length(ST_AsText( ST_SimplifyPreserveTopology(geom, 0.01))) AS sade_agirlik FROM konya.ilce_sinirlari ORDER BY orjinal_agirlik DESC LIMIT 5; -- 📉 2. Haritada karşılaştır (orjinal + sade yan yana) SELECT ad, geom AS orjinal, ST_SimplifyPreserveTopology(geom, 0.01) AS sade FROM konya.ilce_sinirlari LIMIT 5; -- 🌊 3. YUMUŞAT: Köşeleri yuvarla, haritada gör SELECT ad, geom AS orjinal, ST_ChaikinSmoothing(geom, 2) AS yumusak FROM konya.ilce_sinirlari LIMIT 5; -- ⚠️ 4. DİKKAT: 4326'da tolerans derecedir! -- Metre istiyorsan: TR projeksiyonu 5255'e çevir, -- sadeleştir (1000 m), tekrar 4326'ya dön SELECT ad, geom, ST_Transform( ST_Simplify(ST_Transform(geom, 5255), 1000), 4326) AS metrik_1km FROM konya.ilce_sinirlari LIMIT 3;
Simplify mi Smoothing mi Ne Zaman? ▼
📉 Simplify: Harita yavaş, dosya büyük → noktayı azalt, performans kazan.

🌊 Smoothing: Sınırlar testere gibi tırtıklı → görsel olarak yumuşat (sunum/poster için).

Birlikte: Önce Simplify (hızlan), sonra Smoothing (güzelleş) yapılabilir.
Görev: Tolerans Deneyi ▼
1) 1. sorguyu çalıştır — orjinal_agirlik ~6300, sade_agirlik ~1600'e düşer (4 kat hafif).

2) Toleransı değiştir: 0.01 → 0.001 → 0.05. Ağırlık nasıl değişiyor?

3) 2. sorguyu haritada aç — sade sürüm kabalaştı mı, sınır hâlâ tanınıyor mu?

2.2 — ST_Subdivide: Performans Canavarı
06 / 22
Devasa Poligonu Dilimle
Koca bir il sınırı tek parça → indeks işe yaramaz, sorgular yavaş. Çözüm: pizzayı dilimle, malzeme aynı kalsın.
🐌 Sorun: Dev Kutu
Konya tek parça — sınır kutusu tüm ili kaplar. GiST indeksi eleme yapamaz, her sorgu binlerce noktayı tarar.
🍕 Çözüm: ST_Subdivide
1 dev poligon → 5 küçük parça (255 köşe limiti). Her parçanın kutusu minik, indeks anında doğru parçayı bulur.
⚖️ Simplify'dan Farkı
Simplify detay siler (sınır değişir). Subdivide hiç detay kaybetmez — sadece keser. Birleştirince orijinalin aynısı.
💡 Pratik: Subdivide edilmiş tabloya ayrıca GiST indeks kur → JOIN/Intersects sorguları katlarca hızlanır.
-- 🍕 1. Konya'yı parçala — kaç parça çıkıyor? -- 255 = her parçada en fazla 255 köşe SELECT COUNT(*) AS parca_sayisi FROM ( SELECT ST_Subdivide(geom, 255) AS g FROM konya.il_sinirlari WHERE ad ILIKE '%konya%' ) parcalar; -- 🗺️ 2. Parçaları haritada gör (her satır bir dilim) SELECT ST_Subdivide(geom, 255) AS parca FROM konya.il_sinirlari WHERE ad ILIKE '%konya%'; -- ⚖️ 3. Detay korunuyor mu? Parçaları birleştir, -- orijinalle karşılaştır (alan AYNI olmalı) WITH p AS ( SELECT ST_Subdivide(geom, 255) AS g FROM konya.il_sinirlari WHERE ad ILIKE '%konya%' ) SELECT ROUND(ST_Area(ST_Union(g))::numeric, 6) AS parcali_alan, (SELECT ROUND(ST_Area(geom)::numeric, 6) FROM konya.il_sinirlari WHERE ad ILIKE '%konya%') AS orjinal_alan FROM p;
Parçalamak Hızı Neden Artırır? ▼
Büyük şeklin sınırlayıcı kutusu kocaman → GiST indeksi "bu nokta bu ilde mi?" sorusunu eleyemez, binlerce köşeyi tek tek hesaplar.

Subdivide sonrası her parçanın kutusu minik. İndeks önce doğru parçayı saniyede bulur, sadece o küçük parçada hesap yapar. Az nokta = hızlı sorgu.
Görev: Detay Kaybı Var mı? ▼
1) 1. sorgu: Konya kaç parçaya bölündü? (5 bekleniyor)

2) 2. sorguyu haritada aç — il sınırının nasıl dikey/yatay kesildiğini gör.

3) 3. sorgu: parcali_alan ile orjinal_alan birebir aynı olmalı. Subdivide hiçbir detay kaybetmez — Simplify'dan en büyük farkı bu.

2.2 — Doğrusal Referanslama (Linear Referencing)
07 / 22
Yolu Cetvel Gibi Kullan
Bir yol = metre cetveli. "Bu nokta yolun kaçıncı metresinde?" ya da "Yolun %50'si nerede?" — iki yön.
📍→📏 ST_LineLocatePoint
Nokta ver → yolun kaçta kaçı (0–1) söylesin. "Bu kaza yolun %41'inde olmuş."
📏→📍 ST_LineInterpolatePoint
Oran ver (0.5) → o noktanın koordinatını söylesin. "Yolun tam ortası şurası."
🎯 Nokta yolda olmak zorunda değil
Araç yoldan 30 m tarlaya uçsa bile, en yakın izdüşümü alınır — yine doğru km bulunur.
💡 Metreye çevir: Oran (0–1) × ST_Length(geom::geography) = başlangıçtan kaç metre.
-- Senaryo: Gülistan Caddesi'nde (id=6) bir kaza ihbarı -- GPS noktası: 32.5548, 37.9315 (yola tam oturmuyor) -- CTE ile noktayı bir kez tanımla, 3 yerde kullan WITH kaza AS ( SELECT ST_GeomFromText('POINT(32.5548 37.9315)', 4326) AS p ) SELECT y.id, -- 1) Hattı + kaza noktasını birlikte haritada gör ST_Collect(y.geom, k.p) AS harita, -- 2) Yolun kaçta kaçında? (yüzde) ROUND((ST_LineLocatePoint(y.geom, k.p) * 100)::numeric, 1) || '%' AS oran_yuzde, -- 3) Başlangıçtan kaç metre? ROUND((ST_LineLocatePoint(y.geom, k.p) * ST_Length(y.geom::geography))::numeric, 1) || ' m' AS mesafe_metre FROM konya.osm_yollar y, kaza k WHERE y.id = 6; -- → %41.0 | 1044.6 m
Nokta Tam Yolun Üzerinde mi Olmalı? ▼
Hayır. ST_LineLocatePoint, noktanın çizgiye en yakın izdüşümünü (dik inişini) bulur, oranı ona göre verir. Kaza yapan araç yoldan 30 m tarlaya uçsa bile otoyoldaki tam kilometresini bulabilirsin. Üstteki sorgu %41.0 / 1044.6 m döndürür.
Görev: Yolun Tam Ortasını Bul ▼
Ters yön: oran ver, koordinat al. Gülistan Caddesi'nin tam ortasını (0.5) bul ve haritada gör:
SELECT id, ST_LineInterpolatePoint(geom, 0.5) AS orta_nokta FROM konya.osm_yollar WHERE id = 6;
Dene: 0.5 yerine 0.0 (başlangıç) ve 1.0 (son) ver — nokta yolun uçlarına gider.

2.2 — ST_ClosestPoint ve ST_Azimuth
08 / 22
Yön ve Dikme
Bir noktadan yola dik mesafe ve iki nokta arası pusula açısı.
🧭 ST_Azimuth
İki nokta arasındaki pusula yönü. Kuzey = 0°, Doğu = 90°. Sonuç radyan gelir — degrees() ile çevir.
Alaaddin → Mevlana: ≈ 98° (neredeyse tam doğu)
📐 ST_ShortestLine
Bir noktadan en yakın yola çizilen dik çizgi. Noktanın yolun üzerinde olması şart değil — dikme otomatik bulunur.
Örnek: Bina → Gürsesler Sokak ≈ 19.8 m
📌 ST_ClosestPoint
ST_ShortestLine'ın bitiş noktası = çizgi üzerindeki en yakın koordinat. Bina → yol kesişimi için kullanılır.
💡 <-> KNN operatörü, GiST indeksini kullanarak en yakın yolu tablonun tamamını taramadan buluyor. LIMIT 1 ile tek satır döndürülür.
-- AZIMUTH: İki nokta arası bearing (kuzey=0°, doğu=90°) SELECT degrees(ST_Azimuth( ST_SetSRID(ST_Point(32.4920, 37.8727), 4326), -- Alaaddin Tepesi ST_SetSRID(ST_Point(32.5040, 37.8710), 4326) -- Mevlana Müzesi )) AS bearing_derece; -- → 98.06° -- CLOSEST POINT + ShortestLine: Bina → en yakın yol analizi WITH bina AS ( SELECT ST_SetSRID(ST_Point(32.5010, 37.8940), 4326) AS p ) SELECT y.id, -- DBeaver'da Bina noktasını, Yolu ve aralarındaki Dik Çizgiyi tek seferde görelim ST_Collect(ARRAY[b.p, y.geom, ST_ShortestLine(y.geom, b.p)]) AS harita, ROUND(ST_Length(ST_ShortestLine(y.geom, b.p)::geography)::numeric, 1) || ' m' AS uzaklik_metre FROM konya.osm_yollar y, bina b ORDER BY y.geom <-> b.p -- KNN — en yakın 1 yol LIMIT 1; -- → id=4423 | Gürsesler Sokak | 19.8 m
Kritik Soru: Azimuth nerede kullanılır? ▼
Parsel yönelim (cephe) analizlerinde, yoldaki tabelaların bakış yönünü hesaplamada, rüzgâr/güneş etüdünde kullanılır. degrees() ile radyan→derece dönüşümü unutulmamalı.
Görev: En Yakın Yola Dikme ▼
Bir binanın merkezinden en yakın yolun üzerine düşen dik mesafeyi ST_ShortestLine ile çizin ve ::geography ile metreye çevirin.
-- CEVAP: WITH bina AS (SELECT ST_SetSRID(ST_Point(32.5010, 37.8940), 4326) AS p) SELECT ST_AsText(ST_ShortestLine(y.geom, b.p)) AS dik_cizgi, ROUND(ST_Length(ST_ShortestLine(y.geom, b.p)::geography)::numeric, 2) || ' m' AS dik_mesafe FROM konya.osm_yollar y, bina b ORDER BY y.geom <-> b.p LIMIT 1;

2.3 — GiST ve SP-GIST İndeksleme
09 / 22
GiST vs SP-GiST
Her geometri tipi için doğru indeksi seç — ikisi de mekansal ama çalışma mantıkları farklı.
📦 GiST (R-Tree) — Klasör Mantığı
Geometrileri kapsayan Bounding Box'ları ağaç şeklinde üst üste koyar. Kutular birbirinin içine girebilir (iç içe klasörler). Poligon ve LineString için zorunludur.
🔲 SP-GiST (Quad-tree) — Satranç Tahtası
Mekanı kesişmeyen hücrelere böler (satranç tahtası gibi). Noktalar kesişmez → çok daha derin ağaç olmaz → Point (POI, adres) sorgularında GiST'ten hızlı olabilir.
📏 Kural: Poligon/LineString → GIST | Point → SPGIST
Az veri (binler) → PostgreSQL yine de Seq Scan seçebilir. İndeksin farkı milyonlarca satırda görülür.
-- 1) GiST (R-Tree): Poligon ve Çizgiler için mecburidir. (İç içe geçen kutular mantığı) CREATE INDEX IF NOT EXISTS idx_binalar_gist ON konya.osm_binalar USING GIST(geom); -- 2) SP-GiST (Quad-tree): Birbiriyle kesişemeyen Noktalar (POI, Adres) için çok daha hızlıdır! CREATE INDEX IF NOT EXISTS idx_poi_spgist ON konya.poi USING SPGIST(geom);
Kritik Soru: Neden her şeye sadece GiST kurup geçmiyoruz? ▼
GiST, objeleri kapsayan 'Bounding Box'ları üst üste binebilen klasörler gibi tutar; geniş poligonlar için mecburidir. SP-GiST ise mekanı satranç tahtası gibi kesişmeyen hücrelere böler. Milyonlarca POI (Nokta) verisinde SP-GiST kullanmak ağaç derinliğini azalttığı için çok daha hızlı arama sağlar.
Görev: İndeks Kapışması (Performance Test) ▼
konya.poi tablosundaki mevcut GiST indeksini (idx_poi_geom) silin. Yerine SPGIST indeksi kurarak geniş bir alanda nokta sorgusu çalıştırın ve EXPLAIN ANALYZE ile süreyi kıyaslayın.
-- CEVAP (Sırayla Çalıştırın): -- 1. Mevcut GiST indeksini uçurun DROP INDEX IF EXISTS konya.idx_poi_geom; -- 2. Yerine SP-GiST kurun CREATE INDEX idx_poi_spgist ON konya.poi USING SPGIST(geom); -- 3. Hızı EXPLAIN ANALYZE ile test edin EXPLAIN ANALYZE SELECT id, ad, kategori FROM konya.poi WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326);
Dikkat: Neden "Seq Scan" Yaptı? ▼
Eğer analizde Index Scan yerine Seq Scan görüyorsanız hata yapmadınız! PostgreSQL, veri sayısı çok az olduğunda (örn. 1000 satır) indeks ağacını dolaşmanın maliyetli olduğunu bilir ve tabloyu baştan sona okur. İndeksin gücü milyonlarca veride ortaya çıkar. (Zorlamak için: SET enable_seqscan = OFF;)

2.3 — Bounding Box Operatörleri (&&)
10 / 22
Kutu Matematiği
&& operatörü ile milisaniyeler içinde Bounding Box (Sınırlayıcı Kutu) kesişimi tespiti.
-- ST_MakeEnvelope ile (MinX, MinY, MaxX, MaxY) bir kutu oluşturuyoruz. -- && operatörü bu sanal kutu ile kesişen mahalleleri anında bulur. SELECT id, ad FROM konya.mahalleler WHERE geom && ST_MakeEnvelope(32.40, 37.80, 32.60, 37.90, 4326);
Kritik Soru: && neden vazgeçilmezdir? ▼
İndeksler (GiST) arka planda şekillerin kıvrımlarını değil, sadece bu Bounding Box'ları anlar. Gerçek poligon matematiği yapan ST_Intersects komutuna göre binlerce kat daha az yorar ve hızlı çalışır.
Görev: Kutu Sorgusu Yap ▼
Kendi belirlediğiniz bir X,Y aralığındaki binaları (konya.osm_binalar) sadece && ile sorgulayıp hızını test edin.
-- CEVAP: İkisini de EXPLAIN ANALYZE ile çalıştırıp "Execution Time" farkına bakın! -- 1. Sistemleri çökerten o meşhur "Mesafe Tuzağı" nedir ve ondan nasıl kaçınılır?
-- 1) YANLIŞ YÖNTEM: ST_Distance (Sistemi Yorar) -- Mantığı: "Konya'daki MİLYONLARCA bina ile aramdaki mesafeyi TEK TEK hesapla, sonra 500'den küçükleri getir." WHERE ST_Distance(geom, p) < 500 -- 2) DOĞRU YÖNTEM: ST_DWithin (İndeksi Uçurur) -- Mantığı: "Etrafıma anında 500 metrelik sanal bir Bounding Box çiz, kutu dışındakileri hesaplamadan çöpe at!" WHERE ST_DWithin(geom, p, 500)
Kritik Soru: İkisi de aynı sonucu veriyorsa neden ST_Distance yavaş? ▼

ST_DWithin ise arka planda önce o sihirli && (Kutu Kesişimi) operatörünü çalıştırır ve verilerin %99'unu hiç hesap yapmadan elediği için saliseler içinde sonuç verir (Index Scan).

2.5 — VACUUM ve Bloat Yönetimi
18 / 22
Ölü Satırlar (Bloat)
PostgreSQL'de bir satırı güncellediğinizde veya sildiğinizde o satır diskten silinmez! Sadece üstü kırmızı kalemle "ÖLÜ" diye çizilir (MVCC mantığı). VACUUM ise veritabanının temizlikçisidir; bu ölü satırları çöpe atıp yerine yeni verilerin gelmesini sağlar.
VACUUM (Sessiz Temizlikçi)
Arka planda sessizce çalışır, tabloyu kilitlemez. Ölü alanları yeni verilere hazır hale getirir ama tablonun diskteki boyutunu küçültmez.
VACUUM FULL (Yıkım Ekibi)
Tabloyu dışarıya kilitler ve sıfırdan yeniden yazar. Diskteki alanı kesin olarak küçültür ama saatlerce sürebilir, canlı sistemde asla gündüz yapılmaz!
-- 1. Tablolarımızdaki "Ölü Satır" (Dead Tuple) miktarını görelim: SELECT schemaname AS sema_adi, relname AS tablo_adi, n_live_tup AS canli_satir, n_dead_tup AS olu_satir, ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 1) AS olu_yuzde FROM pg_stat_user_tables WHERE schemaname = 'konya' ORDER BY olu_yuzde DESC NULLS LAST; -- 2. Standart temizlik yapıp, istatistikleri (ANALYZE) tazeleyelim: VACUUM ANALYZE konya.mahalleler;
Kritik Soru: Autovacuum zaten kendi kendine temizlemiyor mu? ▼
Evet! Normalde Autovacuum arkaplanda sessizce çalışır. Ancak saniyede yüzlerce konum verisi güncellenen bir araç filosu tablonuz varsa, Autovacuum hızınıza yetişemez ve tablonuz çılgınca şişer (Bloat). Böyle durumlarda manuel olarak veya ayarlarla oynayarak müdahale etmeniz gerekir.
Görev: Ölüleri Say ▼
Yukarıdaki 1. SQL kodunu çalıştırın. Şemamızdaki hangi tablonun ne kadar ölü satır barındırdığını ve yüzde (%) olarak şişme oranını görün.

2.5 — Tablo Bölümleme (Partitioning)
19 / 22
Çekmece Mantığı
Milyonlarca veriyi tek bir devasa tabloda tutmak yerine, arka planda mantıksal çekmecelere (Partition) bölme sanatı.
-- 1. ANA DOLAP (Parent Table): İçinde veri tutmaz, sadece yönlendiricidir. CREATE TABLE konya.binalar_bolunmus ( id SERIAL, geom GEOMETRY(MULTIPOLYGON, 4326), ilce TEXT NOT NULL ) PARTITION BY LIST (ilce); -- 2. ÇEKMECELER (Partitions): Verinin fiziksel olarak yazıldığı gerçek alt tablolar. CREATE TABLE konya.binalar_selcuklu PARTITION OF konya.binalar_bolunmus FOR VALUES IN ('SELÇUKLU', 'Selçuklu'); CREATE TABLE konya.binalar_meram PARTITION OF konya.binalar_bolunmus FOR VALUES IN ('MERAM', 'Meram'); CREATE TABLE konya.binalar_karatay PARTITION OF konya.binalar_bolunmus FOR VALUES IN ('KARATAY', 'Karatay'); -- HAYAT KURTARAN BİLGİ: Beklenmeyen bir ilçe gelirse sistem çökmesin diye "Diğer" çekmecesi açılır. CREATE TABLE konya.binalar_diger PARTITION OF konya.binalar_bolunmus DEFAULT; -- 3. TEST EDELİM: GERÇEK VERİ DOLDURUYORUZ (Mekansal Kesişim ile) -- İlçe sınırlarından "Selçuklu" poligonunu bulur, içindeki tüm binaları keser (ST_Intersects) -- ve Ana Dolaba ekleriz. PostgreSQL bunu otomatik olarak 'binalar_selcuklu' çekmecesine koyar! INSERT INTO konya.binalar_bolunmus (geom, ilce) SELECT b.geom, 'Selçuklu' FROM konya.osm_binalar b JOIN konya.ilce_sinirlari i ON ST_Intersects(b.geom, i.geom) WHERE i.ad ILIKE 'selçuklu';
Kritik Soru: İlçe bazlı Partitioning neden efsanedir? ▼
Siz kodunuzda tek bir tabloymuş gibi (SELECT * FROM binalar_bolunmus WHERE ilce='Selçuklu') sorgu atarsınız. PostgreSQL arka planda sadece Selçuklu çekmecesini açar, Meram ve Karatay dosyalarına hiç dokunmaz (Buna Partition Pruning denir). Ayrıca her çekmecenin indeksi kendine ait ve ufak olduğu için aramalar ışık hızında gerçekleşir.
Görev: Çekmeceyi Doldur ▼
Yukarıdaki kodları çalıştırarak parçalı tablonuzu kurun. Ardından kendi oluşturduğunuz Point verilerini veya mevcut ilçelerdeki binaları bu ana tabloya (INSERT) edin ve verilerin doğru alt tablolara gidip gitmediğini DBeaver'dan kontrol edin!

Gereksizi Ayıkla
Partition Pruning (Bölüm Ayıklama): PostgreSQL'in aranan veri için sadece ilgili çekmeceyi açıp, ilgisiz çekmecelere diskte hiç dokunmaması sanatıdır.
-- 1. PostgreSQL'de bu harika özellik genelde açıktır, emin olmak için: SET enable_partition_pruning = on; -- 2. Ana dolaba (binalar_bolunmus) Selçuklu'yu soralım: EXPLAIN ANALYZE SELECT * FROM konya.binalar_bolunmus WHERE ilce = 'Selçuklu'; -- Çıktıda sadece "binalar_selcuklu" tablosunun okunduğunu, -- "binalar_meram" ve "binalar_karatay"ın tamamen es geçildiğini göreceksiniz.

2.5 — Tablespace ile Disk Yönetimi
21 / 22
Fiziksel Yer Kontrolü
Tablespace = PostgreSQL'in tablo verilerini fiziksel olarak hangi diske yazacağını belirleyen yer. Sık kullanılan tabloları SSD'ye, arşiv tablolarını HDD'ye yönlendirebilirsiniz.
Tipik Senaryolar
• Sık erişilen osm_binalar → SSD
• Geçmiş log_arsiv → HDD
• Ağır indeksler ayrı diskte → I/O paralelliği
-- 1. Tablespace oluştur (Klasör önceden oluşturulmuş olmalı) -- DİKKAT: Docker kullanıyorsanız bu yol konteyner içindeki bir yol olmalıdır. CREATE TABLESPACE ts_hizli LOCATION '/var/lib/postgresql/data/ssd'; CREATE TABLESPACE ts_arsiv LOCATION '/var/lib/postgresql/data/hdd'; -- 2. Mevcut tabloyu SSD'ye taşı (tablo süresince kilitlenir) ALTER TABLE konya.osm_binalar SET TABLESPACE ts_hizli; -- 3. Yeni tabloyu doğrudan belirli tablespace'te oluştur CREATE TABLE konya.log_arsiv ( id BIGSERIAL, islem TEXT, tarih TIMESTAMPTZ ) TABLESPACE ts_arsiv; -- 4. İndeksi tablodan FARKLI bir tablespace'te tut (I/O paralelliği) CREATE INDEX idx_binalar_geom ON konya.osm_binalar USING GIST(geom) TABLESPACE ts_hizli; -- 5. Tablespace listesi ve boyutları SELECT spcname, pg_size_pretty(pg_tablespace_size(oid)) AS boyut, pg_tablespace_location(oid) AS yol FROM pg_tablespace;
Kritik Soru: İndeks ile tablo aynı tablespace'te mi olmalı? ▼
Hayır — aksine farklı disklere koymak I/O paralelliği sağlar. Tablo bir SSD'den okunurken indeks başka bir SSD'den okunabilir. Aynı diske koymak darboğaz oluşturur.
Görev: Aktif/Pasif Ayrımı ▼
Belediye veritabanınızdaki tabloları "sıcak" (sık güncellenen — adresler, binalar) ve "soğuk" (arşiv — eski log'lar) olarak ayırın. Sıcak olanları SSD tablespace'ine taşımayı planlayın.
