PostGIS Akademi — Bölüm 0

SQL'e Giriş

PostGIS'e başlamadan önce tamamlanması önerilen SQL temelleri. SELECT'ten JOIN'e, toplama fonksiyonlarından veri yönetimine — 7 adımda pratik SQL yetkinliği.

Açılış Sorusu

Hiç SQL yazmamış biri, bu bölümün sonunda PostGIS sorgularını okuyup anlayabilir mi?

Yanıt

Evet. SELECT, WHERE, GROUP BY ve JOIN — PostGIS sorgularının %90'ında kullanılan yapılar — bu bölümün konusudur. Geometri sözdizimi yeni olacak, ancak SQL iskeleti tamamen tanıdık gelecek.

Bu Bölümü Kimin Alması Gerekir?

Excel'de çalışmış ancak SQL yazmamış katılımcılar; veya SQL'i tazelemek isteyenler. Temel SQL bilgisi olanlar doğrudan Bölüm 1'e geçebilir.

Alt Modül 0.1 — SELECT ve Veri Modeli

Tablo Yapısı ve SELECT ile Veri Çekme

Açılış Sorusu

Bir veritabanı tablosu ile Excel sayfası arasındaki en kritik yapısal fark nedir?

Yanıt

Her satır PRIMARY KEY ile eşsiz biçimde tanımlanır; hücreler doğrudan düzenlenemez — yalnızca SQL komutlarıyla değiştirilebilir. Bu kısıtlama veri bütünlüğünü ve eş zamanlı erişimi mümkün kılar.

Endüstriyel Bağlam

İBB kentsel dönüşüm birimi 39 ilçenin parsel ve bina verilerini TKGM veritabanından SELECT sorgusuyla çekerek aylık yoğunluk raporları üretmektedir. 2.3 milyon satır saniyeler içinde filtrelenebilir; aynı veriyi Excel'de açmak mümkün değildir.

Kavram

Tablo = satır + sütun + veri tipi kısıtı. SELECT * tüm sütunları, SELECT sütun1, sütun2 seçili sütunları getirir. AS ile alias tanımlanır. SELECT hesaplama yapabilir ama tabloyu değiştirmez.

Örnek

SELECT ad, nufus, nufus / alan_km2 AS yogunluk FROM sehirler ORDER BY yogunluk DESC;

Kritik Soru

SELECT içindeki nufus / alan_km2 AS yogunluk hesabı tabloya kalıcı yeni sütun ekler mi?

Yanıt

Hayır. SELECT içindeki hesaplamalar yalnızca sorgu sonucu için geçerlidir; fiziksel tablo değişmez. Kalıcı sütun için ALTER TABLE ... ADD COLUMN veya CREATE VIEW gerekir.

Bağımsız Çalışma

Nüfusu 1 milyonun üzerindeki şehirleri büyükten küçüğe sırala; yalnızca ad ve nüfus sütunlarını göster.

Beklenen Çıktı

SELECT ad, nufus FROM sehirler WHERE nufus > 1000000 ORDER BY nufus DESC;

Alt Modül 0.2 — Veri Tipleri

PostgreSQL Veri Tipleri ve Doğru Seçim

Açılış Sorusu

Finansal veri için neden TEXT veya REAL yerine NUMERIC kullanılmalıdır?

Yanıt

TEXT sayısal işlemlere izin vermez. REAL (kayan nokta) yuvarlama hatası üretir: 0.1 + 0.2 = 0.30000000000000004. NUMERIC(15,2) kesin ondalık hesap sağlar — muhasebe sistemleri için zorunludur.

Endüstriyel Bağlam

TÜİK nüfus değerlerini INTEGER ile saklar — REAL'in yuvarlama hatası istatistik tutarsızlığı yaratır. İBB acil olay sistemi TIMESTAMPTZ kullanır: UTC'de saklanan zaman İstanbul ve Avrupa ofisleri için otomatik yerel saate çevrilir.

Kavram

Tam sayı: INTEGER / BIGINT / SERIAL. Ondalık: NUMERIC(p,s) (kesin), REAL (yaklaşık). Metin: TEXT (tercih), VARCHAR(n). Zaman: DATE, TIMESTAMPTZ. Diğer: BOOLEAN, UUID, JSONB. PostGIS: geometry / geography.

Örnek

CREATE TABLE sehirler (id SERIAL PRIMARY KEY, ad TEXT NOT NULL, nufus INTEGER, alan_km2 NUMERIC(10,2), kurulis DATE);

Kritik Soru

Coğrafi koordinatlar (enlem/boylam) için hangi tip kullanılmalıdır?

Yanıt

Ayrı NUMERIC sütunlar yerine PostGIS geometry(POINT, 4326) tipi. Mekansal indeksleme, projeksiyon dönüşümü ve mesafe hesabı geometry tipiyle doğrudan yapılır — iki ayrı sayı sütunu bu özellikleri desteklemez.

Bağımsız Çalışma

Bir acil olay tablosu tasarla: id, aciklama, onem_derecesi, kayit_zamani, kapandi_mi sütunları için doğru veri tiplerini seç.

Beklenen Çıktı

id BIGSERIAL, aciklama TEXT, onem_derecesi SMALLINT, kayit_zamani TIMESTAMPTZ, kapandi_mi BOOLEAN DEFAULT false. Geometri sütunu için: konum geometry(POINT, 4326).

Alt Modül 0.3 — WHERE ile Filtreleme

Koşullu Filtreleme: Operatörler ve Mantıksal Bağlaçlar

Açılış Sorusu

10 milyonluk bir tabloda WHERE ile filtreleme tüm tabloyu okumak zorunda mı?

Yanıt

İndeksli sütunda hayır. Veritabanı indeks ağacından ilgili sayfaları doğrudan bulur; tüm tabloyu taramaz. PostGIS'te geometri sütunları için GiST indeksi bu görevi üstlenir.

Endüstriyel Bağlam

Kadastro sistemleri WHERE koşuluyla ilçe ve ada-parsel bazlı filtreleme yaparak sorgu süresini %95 oranında kısaltır. Günlük on binlerce sorgu bu sayede milisaniyeler içinde yanıt verir; indekssiz aynı sorgu dakikalar alırdı.

Kavram

Karşılaştırma: = != > < >= <=. Mantıksal: AND OR NOT. Özel: LIKE 'İst%', IN ('Ankara','İzmir'), BETWEEN 1 AND 5, IS NULL / IS NOT NULL.

Örnek

SELECT ad, nufus FROM sehirler WHERE nufus > 1000000 AND bolge IN ('Marmara','Ege') AND ad LIKE 'İ%';

Kritik Soru

WHERE nufus > 1000000 OR bolge = 'Ege' ile AND arasındaki sonuç farkı nedir?

Yanıt

OR: iki koşuldan biri yeterliyse satır dahil edilir — nüfusu 1M+ VEYA Ege bölgesindeki tüm şehirler. AND: her iki koşul aynı anda sağlanmalıdır — nüfusu 1M+ VE Ege bölgesinde (çok daha küçük küme).

Bağımsız Çalışma

Adı 'A' ile başlayan VEYA nüfusu 500.000'in altındaki şehirleri listele. NULL nüfuslu kayıtları hariç tut.

Beklenen Çıktı

SELECT * FROM sehirler WHERE (ad LIKE 'A%' OR nufus < 500000) AND nufus IS NOT NULL;

Alt Modül 0.4 — Sıralama ve Limit

ORDER BY, LIMIT, OFFSET ve DISTINCT

Açılış Sorusu

Nüfusa göre en büyük 5 şehri bulmak için SQL'in en kısa yolu nedir?

Yanıt

SELECT ad, nufus FROM sehirler ORDER BY nufus DESC LIMIT 5; — ORDER BY büyükten küçüğe sıralar, LIMIT ilk 5'i alır. İndekssiz ORDER BY büyük tablolarda pahalı olabilir; indeks seçimi kritiktir.

Endüstriyel Bağlam

AFAD, deprem anında en yüksek nüfus yoğunluğuna sahip ilçeleri ORDER BY yogunluk DESC LIMIT 10 ile belirleyerek kaynak tahsisini otomatik önceriklendirir. Web uygulamalarında sayfalama LIMIT + OFFSET ile oluşturulur.

Kavram

ORDER BY sütun DESC/ASC — çok sütun: ORDER BY bolge ASC, nufus DESC. LIMIT n: ilk n satır. OFFSET m: ilk m'i atla. DISTINCT: tekrarlayan değerleri kaldırır, her değerden bir örnek döndürür.

Örnek

SELECT DISTINCT bolge FROM sehirler ORDER BY bolge ASC;
SELECT ad, nufus FROM sehirler ORDER BY nufus DESC LIMIT 5 OFFSET 10;

Kritik Soru

ORDER BY büyük tablolarda neden pahalı olabilir?

Yanıt

Sıralama O(n log n) karmaşıklığında; milyonlarca satırda bellek yetersizse geçici disk yazımı (temp file) devreye girer. Çözüm: sıralama sütununa B-tree indeks. Mekansal sorgularda ORDER BY geom <-> nokta için GiST indeksi gerekir.

Bağımsız Çalışma

Nüfusça 3. ile 7. sıradaki şehirleri listele (baştan 2'yi atla, 5 tane getir).

Beklenen Çıktı

SELECT ad, nufus FROM sehirler ORDER BY nufus DESC LIMIT 5 OFFSET 2; → 3.–7. büyük şehirler sıralı olarak gelir.

Alt Modül 0.5 — Toplama Fonksiyonları ve Gruplama

COUNT, SUM, AVG — GROUP BY ve HAVING

Açılış Sorusu

Her bölgenin ortalama nüfusunu tek sorguda görmek için ne gerekir?

Yanıt

SELECT bolge, AVG(nufus) FROM sehirler GROUP BY bolge; — GROUP BY satırları bölgeye göre kümelere ayırır, her küme için AVG ayrı hesaplanır. PostGIS'te: ilçe bazlı toplam alan hesabı için aynı yapı kullanılır.

Endüstriyel Bağlam

Sağlık Bakanlığı, GROUP BY il ile her ildeki yatak sayısını topluyor; HAVING SUM(yatak) < 500 ile yetersiz kapasiteli illeri otomatik olarak kritik listesine alıyor. TÜİK bölgesel istatistik bültenlerini aynı yapıyla üretiyor.

Kavram

Fonksiyonlar: COUNT(*) (NULL dahil), COUNT(sütun) (NULL hariç), SUM, AVG, MIN, MAX. HAVING: gruplama sonrası filtre (WHERE satır filtreler, HAVING grup filtreler). SQL sırası: FROM→WHERE→GROUP BY→HAVING→SELECT→ORDER BY.

Örnek

SELECT bolge, COUNT(*) AS sehir_sayisi, AVG(nufus)::INT AS ort_nufus FROM sehirler GROUP BY bolge HAVING COUNT(*) > 2 ORDER BY ort_nufus DESC;

Kritik Soru

Neden WHERE COUNT(*) > 2 yazılamaz?

Yanıt

WHERE, GROUP BY'dan önce çalışır; bu aşamada henüz gruplama yapılmamıştır — COUNT değeri bilinmez. Gruplama sonrası filtre için HAVING kullanılır. Bu SQL çalışma sırasının doğal bir sonucudur.

Bağımsız Çalışma

İlçe başına toplam bina sayısını hesapla; 50'den az binası olan ilçeleri hariç tut. Sonuçları büyükten küçüğe sırala.

Beklenen Çıktı

SELECT ilce, COUNT(*) AS bina_sayisi FROM binalar GROUP BY ilce HAVING COUNT(*) >= 50 ORDER BY bina_sayisi DESC;

Alt Modül 0.6 — JOIN ile Tablo İlişkileri

INNER JOIN, LEFT JOIN ve Mekansal JOIN

Açılış Sorusu

Şehir bilgileri bir tabloda, bölge detayları başka tabloda saklanıyorsa ikisini nasıl birleştirirsiniz?

Yanıt

FROM sehirler s JOIN bolgeler b ON s.bolge_id = b.id. PostGIS'te en sık kullanılan JOIN: JOIN ilceler i ON ST_Within(bina.geom, i.geom) — birleştirme koşulu PRIMARY KEY değil geometrik ilişkidir.

Endüstriyel Bağlam

TKGM'de parsel-mülk sahiplik zinciri 4 tabloyu JOIN ederek sorgulanmaktadır: parsel → ada → ilçe → il. PostGIS'te 5. tablo geometri içerir; mekansal JOIN (ST_Intersects) ile bağlanır. Tüm zincirleme JOIN'ler milisaniyeler içinde yanıt verir.

Kavram

INNER JOIN: her iki tabloda eşleşme varsa satır dahil. LEFT JOIN: sol tablodaki tüm satırlar; sağda eşleşme yoksa NULL. CROSS JOIN: kartezyen çarpım. PostGIS LATERAL: per-row alt sorgu, en yakın tesis bulmada kullanılır.

Örnek

SELECT s.ad, b.bolge_adi, s.nufus FROM sehirler s JOIN bolgeler b ON s.bolge_id = b.id WHERE s.nufus > 500000;

Kritik Soru

LEFT JOIN ile INNER JOIN ne zaman aynı sonucu verir?

Yanıt

Sol tablodaki her satırın sağ tabloda mutlaka bir eşleşmesi olduğunda. Hiçbir NULL gelmeyecekse INNER = LEFT. Uygulamada: her şehrin bir bölge kaydı varsa ikisi özdeştir — ancak bu durum her zaman garanti edilemez.

Bağımsız Çalışma

Her bölgenin en kalabalık şehrini listele. (İpucu: GROUP BY + MAX veya alt sorgu gerekir.)

Beklenen Çıktı

SELECT b.bolge_adi, s.ad, s.nufus FROM sehirler s JOIN bolgeler b ON s.bolge_id = b.id WHERE s.nufus = (SELECT MAX(nufus) FROM sehirler s2 WHERE s2.bolge_id = s.bolge_id);

Alt Modül 0.7 — Veri Yönetimi (DML)

INSERT, UPDATE, DELETE ve CREATE TABLE

Açılış Sorusu

SQL yalnızca veri okumak için mi kullanılır? Gerçek zamanlı olay kaydı nasıl yapılır?

Yanıt

Hayır. INSERT yeni kayıt ekler, UPDATE mevcut kaydı değiştirir, DELETE siler. Gerçek zamanlı sistemlerde her yeni olay INSERT ile anlık yazılır. PostGIS'te geometri sütununa: ST_SetSRID(ST_MakePoint(lon, lat), 4326) değeri eklenir.

Endüstriyel Bağlam

112 Acil Yardım sistemi her çağrıyı saniyeler içinde INSERT ile kaydeder; aktif çağrıların koordinatları UPDATE ile güncellenir; kapanan vakalar DELETE değil arşiv tablosuna INSERT + kapandi = true UPDATE ile işaretlenir — veri kaybedilmez.

Kavram

INSERT INTO tablo(sütunlar) VALUES(...); — çok satır aynı anda. UPDATE tablo SET sütun = değer WHERE koşul; — WHERE olmadan tüm tablo güncellenir. DELETE FROM tablo WHERE koşul; — WHERE olmadan tüm tablo silinir. Önce SELECT ile etkilenecek satırları görün.

Örnek

INSERT INTO sehirler(ad, nufus, bolge_id) VALUES('Muğla', 1020000, 3);
UPDATE sehirler SET nufus = 1050000 WHERE ad = 'Muğla';

Kritik Soru

UPDATE'de WHERE koşulu yazılmazsa ne olur? Nasıl önlenir?

Yanıt

Tablodaki tüm satırlar güncellenir — geri dönüşü zordur. Önlem: işlemi BEGIN; ile başlat, UPDATE'den önce SELECT ile etkilenecek satırları gör, onaylarsan COMMIT;, hatalıysa ROLLBACK; ile iptal et.

Bağımsız Çalışma

Yeni bir şehir ekle, nüfusunu güncelle, ardından silmeden önce SELECT ile doğrula. Tüm işlemi transaction içinde yap.

Beklenen Çıktı

BEGIN; INSERT INTO sehirler(ad,nufus) VALUES('Test',100); UPDATE sehirler SET nufus=200 WHERE ad='Test'; SELECT * FROM sehirler WHERE ad='Test'; ROLLBACK; → Test şehri eklenip silinir, tablo etkilenmez.
